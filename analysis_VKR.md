## 🔴 ОШИБКИ (критические несоответствия)

### Ошибка 1 — Листинг 1 (раздел 3.1) не соответствует реальному коду

**Документ (строка ~1527–1554)** показывает упрощённую версию `startCapture`, в которой:
- Поиск устройства происходит **только по имени или IP** (одним циклом)
- Если не найдено — сразу возвращается `false` (неявно, через `!device_->open()`)

**Реальный код** (`TrafficMonitor.cpp`, строки 2619–2719) реализует **двухэтапную нечёткую стратегию**:
1. Сначала поиск по точному имени/IP
2. Затем поиск по очищенному описанию с приоритизацией физических устройств над виртуальными (Hyper-V, Microsoft NDIS)

Листинг в документе **намеренно упрощён** (что допустимо), но текст в разделе 3.1 явно заявляет о «нечёткой двухэтапной стратегии поиска» — а листинг её не демонстрирует. **Нужно использовать реальный код или указать, что листинг упрощён.**

**Документ (строка ~1911):**
```cpp
if (result.PPS < NOISE_THRESHOLD_PPS) {
```

**Реальный код** (`DetectionEngine.cpp`, строка 237):
```cpp
if (result.pps < NOISE_THRESHOLD_PPS) {
```

В документе поле написано как `PPS` (верхний регистр), в реальном коде — `pps` (нижний регистр). В таблице БД (табл. 2.8) поле также называется `PPS`. Несоответствие имён в коде и документе. Нужно проверить структуру `DetectionResult` в `Protocol.hpp` и использовать единообразное написание.

### Ошибка 3 — Блок-схема алгоритма блокировки (рис. 2.12) содержит логическую инверсию

**Документ (строка ~1261–1263):**
```
CheckIPValid -- Да --> LogError ["Журналирование ошибки"]
CheckIPValid -- Нет --> CheckAlreadyBlocked
```

**Логика неверная**: ветка «Да» (IP корректен) должна вести к проверке блокировки, а «Нет» (IP некорректен) — к журналированию ошибки. В документе всё **перепутано**: «Да» (IP корректен) → ошибка, «Нет» (IP некорректен) → продолжение.

**Реальный код** (`FirewallManager.cpp`, строка 28–34):
```cpp
if (addr.isNull()) {  // isNull = IP некорректен → ошибка
    AppLogger::get()->error("Invalid IP format");
    return;
}
// Далее: IP корректен — продолжаем
```

## 🟠 ЗАМЕЧАНИЯ ПО ОФОРМЛЕНИЮ И СТРУКТУРЕ

### Замечание 1 — Дублированные комментарии в листингах приложения Б

В приложении Б (страницы TrafficMonitor.cpp) каждый блок документации **дублируется дважды**:
```
// Назначение: Выполнение логики метода X ...
// Назначение: Выполнение логики метода X ...  ← дубль
```
Также `// --- Начало функционально законченной части ---` встречается **дважды подряд** в каждом методе. Это артефакт автогенерации — нужно удалить дублирование.

### Замечание 2 — Таблица 4.2 не соответствует своему заголовку

**Заголовок** «Состав технических и программных средств испытательного стенда» подразумевает перечень оборудования, но **содержимое** — это таблица сценариев и результатов тестирования. Нужно разделить на две таблицы или исправить заголовок.

### Замечание 3 — Ссылки [N] в содержании не заполнены

По всему документу страницы обозначены `[N]`. Это нормально для черновика, но аннотация также содержит `[N]` для числа страниц — это нужно заполнить перед финальной сдачей.

### Замечание 4 — Раздел 2.6 упоминает `ReportGenerator` с форматом PDF

**Документ (строка ~1405):** «`ReportGenerator` — Формирование детальных отчетов об инцидентах в формате **PDF**»

**Реальный код** (`ReportGenerator.hpp`, 357 байт — очень маленький файл): судя по размеру, реализация минимальная. Стоит проверить, реализован ли PDF экспорт или только CSV.

### Замечание 5 — Описание `DataBridge` в диаграмме монитора

**Документ (диаграмма рис. 2.13):** `DataBridge` содержит `sendBpfConfig(enabled)` и `sendDumpConfig(enabled)`.

**Реальный код** (`DataBridge.hpp`, 1635 байт): нужно проверить соответствие. Класс `DataBridge` в общей структуре — это «прослойка» между GUI и сетью/БД, что верно.

### Замечание 6 — Отсутствует описание класса `PcapDumper`

В коде есть класс `PcapDumper` (`src/network/PcapDumper.hpp`), который используется в `DetectionEngine`. В документе он **нигде не описан** (ни в табл. 2.1, ни в диаграммах). Это нарушение полноты описания.

### Замечание 7 — Диаграмма DetectionEngine не показывает `PcapDumper`

**Документ (диаграмма рис. 2.6):** перечислены поля `monitor_`, `extractor_`, `inferencer_`, `isUnderAttack_`, `isMitigationEnabled_`.

**Реальный код** (`DetectionEngine.hpp`): есть также поле `PcapDumper dumper_`, флаги `isReplayMode_`, `replayFinished_`, `dumpEnabled_`, `pcapDumpDir_`, накапливающие данные поля `maxPps_`, `maxConfidence_`, `attackType_`, `attackSrcIps_`. Диаграмма значительно **неполная**.
