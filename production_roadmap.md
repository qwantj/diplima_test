# План развития проекта (Production-ready Roadmap)

Этот документ объединяет ключевые векторы оптимизации и рефакторинга для перевода системы DDoS-детекции на уровень Production-ready. План разбит на логические уровни от критичных узких мест (CPU/Memory) до высокоуровневой архитектуры и DevOps.

## Уровень 1: Улучшение алгоритмов захвата и обработки трафика (Network & Core)
1. **Реализация 5-tuple Flow Tracking:** Переход от агрегации по интерфейсу к отслеживанию отдельных сессий (Source IP, Dest IP, Source Port, Dest Port, Protocol). Это позволит строить профили конкретных подключений и точечно блокировать злоумышленников (BPF-фильтры).
2. **Оптимизация аллокации памяти (Memory Pooling / Zero-copy):** В `TrafficMonitor::onPacketArrived` происходит копирование пакета. В высоконагруженных сетях это перегрузит аллокатор. Необходимо внедрить Memory Pools или кольцевые буферы с заранее выделенной памятью для сырых байт пакетов и объектов `DetectionResult`.
3. **Замена активного ожидания на `BlockingConcurrentQueue`:** В `DetectionEngine::inferenceLoop` заменить `moodycamel::ConcurrentQueue` на `BlockingConcurrentQueue`, чтобы поток мог аппаратно "засыпать" без нагрузки на CPU и моментально просыпаться.
4. **Отказ от полного парсинга пакетов (`FeatureExtractor`):** Отказаться от `pcpp::Packet` (L2-L7 парсинг) в пользу прямого приведения указателей (C-structs `iphdr*`, `tcphdr*`) для "ленивого" и сверхбыстрого извлечения только нужных заголовков.
5. **Kernel Bypass для перехвата трафика:** Переход с Npcap/libpcap на драйверы DPDK или PF_RING (через `pcpp::DpdkDevice`), либо XDP/eBPF для Linux. Это поднимет лимит обработки до 10-40+ млн пакетов/сек в обход сетевого стека ОС.

## Уровень 2: Оптимизация ML-пайплайна (ModelInferencer)
1. **Аппаратное ускорение (Hardware Acceleration):** Замена дефолтного CPU Execution Provider на TensorRT/CUDA (Nvidia), OpenVINO (Intel) или DirectML (Windows) для радикального снижения задержек.
2. **Батчинг (Batch Inference):** Вместо вызова `session_->Run()` для одного вектора (1xN), накапливать признаки в батчи (например, 64-128 потоков из Flow Tracking) для утилизации SIMD-инструкций.
3. **Квантование моделей (Quantization):** Перевод весов моделей (особенно MLP) из FLOAT32 в FLOAT16 или INT8 средствами ONNX Runtime для снижения потребления памяти и ускорения инференса.
4. **Онлайн-калибровка вероятностей:** Внедрение Platt Scaling или изотонической регрессии, чтобы выходная `Confidence` (вероятность) для MLP-модели была математически обоснованной.

## Уровень 3: База данных и хранение (PostgreSQL)
1. **Отказ от JSON-сериализации:** В `DatabaseManager` заменить конвертацию `features` в строку JSON на использование нативного типа массивов PostgreSQL (`REAL[]`) или бинарного биндинга.
2. **Переход на TimescaleDB:** Использование официального расширения для PostgreSQL для автоматического партиционирования по времени (hyper-tables) и колоссального ускорения записи (INSERT) time-series данных.
3. **Очистка старых данных (Data Retention):** Добавление фонового джоба (или continuous aggregates в TimescaleDB) для удаления/сжатия исторических данных.
4. **Пул соединений (Connection Pooling):** При росте нагрузки использовать `QSqlDatabase::cloneDatabase` с пулом потоков для избежания блокировок БД.

## Уровень 4: Улучшение GUI и визуализации (Qt Monitor)
1. **Аппаратное ускорение графиков (OpenGL):** Для `QLineSeries` на высоких частотах обновления обязательно вызывать `series->setUseOpenGL(true)`.
2. **Децимация данных (Data Decimation):** Реализация алгоритма сжатия (например, LTTB - Largest Triangle Three Buckets) для прореживания данных при отдалении (zoom out) графиков с сохранением визуальных пиков атак.
3. **Разделение логики (MVVM Pattern):** Выделение слоя ViewModel, чтобы виджеты (`DashboardWidget`) только биндились к свойствам, а агрегация данных происходила вне GUI-потока.

## Уровень 5: Архитектура, кодовая база и Сеть
1. **Оптимизация сетевого протокола (DataBridge / TCP):** Переход с JSON/текстовых протоколов (QByteArray) на бинарную сериализацию без копирования памяти (Protocol Buffers, FlatBuffers, Cap'n Proto). Это снизит CPU overhead и сетевой трафик.
2. **Dependency Injection (DI):** Уменьшение жесткой связности между `DetectionEngine`, `FeatureExtractor` и `ModelInferencer` через абстрактные интерфейсы для облегчения Unit-тестирования.
3. **Наблюдаемость (Observability):** Добавление экспортера метрик в формате Prometheus (порт `/metrics`) для мониторинга процесса сбора данных через DevOps-инструменты (Grafana).

## Уровень 6: Автоматизация и DevOps
1. **Тестирование (GTest):** Написание Unit-тестов для критичных узлов: `FeatureExtractor` (правильность извлечения фич) и парсера Protocol.
2. **CI/CD Pipeline:** Настройка GitHub Actions/GitLab CI для сборки проекта под Windows (MSVC) и запуска статических анализаторов (clang-tidy, cppcheck) для выявления утечек памяти или race conditions в lock-free структурах.
