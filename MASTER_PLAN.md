# Единый план восстановления и развития проекта DDoS Detector

Данный документ представляет собой полную компиляцию всех исторических планов, архитектурных решений, отчетов о багах и задач (из каталога `docs/PLANS`), созданных в ходе разработки "Программного комплекса обнаружения атак типа «отказ в обслуживании»" (Студент: Дерюга А.А., А-08-22).

Документ служит единой точкой входа (Single Source of Truth) для понимания эволюции проекта, его архитектуры и принятых технических решений.

---

## 1. Исходное задание и Требования (ВКР)

**Тема:** Разработка программного комплекса обнаружения атак типа «отказ в обслуживании».
**Стек:** C++17, Qt 6, ONNX Runtime, PcapPlusPlus, PostgreSQL, Python (для обучения ML).

**Основные функциональные требования:**
1.  Захват сетевого трафика в режиме реального времени.
2.  Загрузка и анализ ранее сохраненных файлов дампов трафика (PCAP).
3.  Извлечение ключевых признаков из заголовков сетевых пакетов (Feature Extraction).
4.  Классификация потоков с использованием ML-модели.
5.  Сохранение информации об атаках и системных событиях в БД.
6.  Визуализация статистики и результатов в графическом интерфейсе (GUI).

---

## 2. Архитектура Системы (C4 Model)

Система разделена на три основных компонента для обеспечения отказоустойчивости и высокой производительности:

1.  **`ddos_collector.exe` (Сбор и Анализ):** Консольное C++ приложение.
    *   **network**: `TrafficMonitor` (Npcap/PcapPlusPlus захват), `FeatureExtractor` (Извлечение фичей, нормализация), `PcapDumper` (Сохранение `benign.pcap` / `attack.pcap`).
    *   **core**: `DetectionEngine` (Оркестратор: захват -> фичи -> инференс -> результат, окно 2 сек).
    *   **ml**: `ModelInferencer` (ONNX Runtime, классификация).
    *   **common**: `TcpServer` (передача JSON по порту 50050), `DatabaseManager` (PostgreSQL INSERT), `FileBuffer` (Ring-буфер для защиты от потерь при разрыве TCP).
2.  **PostgreSQL (База Данных):** `ddos_detection_db`. Хранит `sessions`, `events`, `stats_snapshots`, `security_events`.
3.  **`ddos_monitor.exe` (Управление и Визуализация):** Qt 6 GUI приложение.
    *   `TcpClient` (JSON стрим), `DataBridge` (мост между TCP/DB и UI).
    *   `DashboardWidget` (PPS графики, Donuts ресурсов).
    *   `LogWidget`, `SessionWidget`, `EventHistoryWidget` (Журнал событий, история).

## 3. Эволюция проекта (History & Walkthroughs)

Проект прошел через несколько масштабных фаз рефакторинга и оптимизации:

### Фаза 1: Оптимизация Производительности (Load Shedding & Lock-Free)
*   **Проблема:** Переполнение RAM (OOM) и высокие задержки при Volumetric DDoS атаках (130k+ PPS).
*   **Решение:**
    *   Внедрение Lock-Free очередей (`moodycamel::ConcurrentQueue`) вместо `std::mutex`.
    *   Использование `std::vector::reserve` и пулов памяти.
    *   **Load Shedding:** Установка `MAX_QUEUE_SIZE = 500000`. При переполнении пакеты отбрасываются (drop), а `drop_rate` передается в монитор.
    *   Оптимизация ONNX: `SetIntraOpNumThreads(1)`, выбор Execution Provider (CPU/DirectML).

### Фаза 2: Решение проблем с GUI и Сетью (PPS Graph Bug & TCP)
*   **Проблема:** Зависание интерфейса монитора на несколько минут при подключении к коллектору, время на графиках "замерзало" на `03:00:00`, цикличное переподключение.
*   **Решение:**
    *   Исправлено усечение (truncation) 64-битного `timestamp` до 32-битного при парсинге JSON.
    *   Отказ от добавления точек по одной (`append`) в цикле из 10,000 элементов. Внедрен **Batching** через `DataBridge` и использование метода `replace()` в `QLineSeries` для мгновенной отрисовки истории.
    *   Графики теперь используют оригинальное время события (`result.timestamp`), а не системное время отрисовки.

### Фаза 3: Интеграция ML и Динамических Признаков (Auto-Discovery)
*   **Проблема:** Изначально система была жестко привязана к 8 признакам и модели `rf_model.onnx`.
*   **Решение:**
    *   **Hot Swap:** Потокобезопасная (`std::shared_mutex`) замена ONNX-сессий на лету через интерфейс монитора.
    *   **Auto-Discovery:** Использование файла `scaler_params.json` для динамического определения порядка и количества признаков.
    *   Поддержка параметра `"use_log1p"` для моделей типа MLP, требующих предварительного логарифмирования признаков перед `StandardScaler`.
    *   Универсальное извлечение вероятности атаки независимо от формата вывода ONNX (Tensor `[1,2]` или Sequence of Maps `skl2onnx`).

### Фаза 4: Улучшение визуализации (Monium / Grafana Style)
*   **Дизайн:** Переход на премиальную антрацитовую тему (Dark/Light/System) через класс `ThemePalette`.
*   **Дашборд:** Добавление интерактивных `InteractiveChartView` (зум колесиком, перемещение).
*   **Инфографика:** Внедрение кольцевых диаграмм (Donuts) для системных метрик (`SystemMetricsCollector`: CPU/RAM) и распределения трафика (TCP/UDP/ICMP).
*   **Сетка здоровья:** Внедрение `AlertGridWidget` (Сетка алертов) и топологических карт.
*   **Инциденты:** Добавление горизонтального таймлайна (`TimelineWidget`) на 24 часа с цветовой индикацией атак.

### Фаза 5: Маркировка и Сохранение Дампов (PCAP)
*   **Проблема:** Необходимость сохранять маркированный трафик без OOM и блокировок I/O.
*   **Решение:**
    *   Реализация `PcapDumper` с потоковой записью во временный `_temp.pcap` файл.
    *   После инференса (каждые 2 сек) данные из `_temp.pcap` аппендятся либо в `benign.pcap`, либо в `attack.pcap`. Временный файл удаляется.

## 4. План Дальнейшего Развития (Recommendations & Roadmap)

На основе архитектурного анализа системы (документ `analysis_results23.md`), предложены следующие направления для развития уровня Enterprise:

1.  **Активное подавление (Active Mitigation) через Windows Firewall API:**
    *   Сейчас BPF-фильтр блокирует IP только на уровне Npcap. Необходимо интегрировать WFP (Windows Filtering Platform) или вызовы `netsh advfirewall` для реального блокирования атакующих на уровне ОС.
2.  **Анализ с отслеживанием сессий (Per-Flow Tracking):**
    *   Переход от "Single Window Aggregator" к 5-Tuple хэш-таблице для накопления фичей по каждому отдельному TCP/UDP соединению. Это позволит ML-модели находить точечные L7-атаки (например, Slowloris) среди легитимного трафика.
3.  **Генерация PDF-отчета об инцидентах:**
    *   Добавление функции формирования Security Report (через `QPrinter` и `QTextDocument`) с графиками, описанием DDoS атаки, Top-IPv4 векторами и вердиктом ONNX модели (отличное дополнение для дипломной работы).
4.  **Расширенные метрики сетевого уровня:**
    *   Сбор статистики по TCP флагам (SYN, ACK, PSH, URG, FIN, RST).
    *   Сбор статистики по размеру окна (TCP Window Size) и энтропии payload-а (для выявления мусорного UDP-флуда).
5.  **Защита БД (Adaptive Aggregation):**
    *   Если модель уверена в DDoS на 99% уже 3 секунды подряд, новые пакеты агрегируются в памяти и сбрасываются в БД только по завершению атаки, защищая I/O диска при Volumetric Floods (частично реализовано в `enqueueSecurityEvent`).
