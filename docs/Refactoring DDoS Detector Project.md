# План рефакторинга проекта обнаружения DDoS-атак

  

Дата актуализации: 08.04.2026

  

## 1. Цель рефакторинга

  

Упростить сопровождение и обеспечить предсказуемое развитие комплекса без регрессий:

  

- четкие границы между захватом, ML, БД и UI;

- минимизация связности между компонентами;

- сохранение совместимости с текущими exe и протоколом.

  

## 2. Целевая модульная структура

  

- src/core: orchestration и контракты данных

- src/network: захват и предварительная агрегация

- src/ml: lifecycle и infer ONNX

- src/common: protocol, tcp, db, logging, metrics

- src/monitor_ui: визуализация и пользовательские сценарии

  

## 3. Ключевые проблемы текущей структуры

  

- Часть runtime-настроек дублируется в коде collector/monitor.

- Протокол и UI-ожидания неявно связаны.

- Документация исторически содержит устаревшие порты/параметры.

  

## 4. План рефакторинга (без ломки API)

  

### Фаза 1. Конфигурация

  

- Ввести RuntimeConfig и единый источник параметров.

- Синхронизировать default-значения между collector и monitor.

  

### Фаза 2. Контракты

  

- Зафиксировать schema полей для stats/snapshot/notify.

- Добавить protocol_version в каждое сообщение.

  

### Фаза 3. Изоляция логики

  

- Выделить отдельный сервисный слой для replay-команд.

- Изолировать код БД от UI-виджетов через DataBridge API.

  

### Фаза 4. Тестируемость

  

- Добавить unit-тесты на Protocol serialize/deserialize.

- Добавить интеграционный тест replay -> DB insert.

  

## 5. Инварианты после рефакторинга

  

- Два исполняемых файла сохраняются: ddos_collector и ddos_monitor.

- PostgreSQL остается обязательной частью комплекса.

- Канал реального времени остается TCP JSON.

  

## 6. Критерии завершения

  

- Все сценарии из QUICK_START выполняются без ручных правок кода.

- Документация и фактические defaults совпадают.

- Нет регрессий по live/replay и записи в БД.
