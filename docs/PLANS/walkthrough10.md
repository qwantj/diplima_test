# Разработка системы сохранения промаркированных PCAP-дампов

## Цель
Реализовать сохранение захватываемых пакетов в файлы `benign.pcap` и `attack.pcap` в зависимости от вердикта нейросети каждые 2 секунды. При этом необходимо избежать:
1.  **Ошибки Out-Of-Memory (OOM):** нельзя хранить миллионы пакетов в буфере RAM.
2.  **Лагов I/O I/O Block:** сохранение на диск не должно блокировать основной поток захвата.
3.  **Спама мелкими файлами:** вместо создания сотен мелких файлов каждые 2 секунды, использовать по одному `benign` и `attack` файлу на сессию.

## Что было сделано

### 1. Реализация потоковой записи `PcapDumper`

Для минимизации потребления RAM была реализована концепция "временного потокового файла":
- **`m_tempWriter`**: `PcapFileWriterDevice`, который стримит сырые пакеты прямо на диск (во временный `.pcap` файл).
- По завершении 2-секундного окна вызывается `finalizeWindow(label)`, которая определяет, к какому типу отнести окно (`benign` или `attack`).
- Все пакеты из временного файла дописываются (appending) в конец нужного сессионного файла.
- Временный файл удаляется и создаётся заново под следующее окно.

> [!TIP]
> **Автоопределение LinkLayer:** Для корректного создания PCAP-файлов необходимо знать LinkLayerType (Ethernet, Loopback, etc.). В `PcapDumper` он не жестко зашит, а определяется динамически по `packet.getLinkLayerType()` на самом первом захваченном пакете. Это решило ошибку "Cannot write a packet with a different link layer type" на Loopback-интерфейсах.

### 2. Интеграция с `DetectionEngine` и CLI

- В класс `DetectionEngine` добавлены вызовы:
  - `m_pcapDumper->writePacket(packet)` в `handlePacket()`
  - `m_pcapDumper->finalizeWindow(label)` после получения предсказания от модели.
- В `collector_main.cpp` добавлен флаг `--pcap-dir`. Указание флага включает функцию дампа и создаёт новую сессионную папку `pcap_dumps/session_YYYYMMDD_HHMMSS`.

## Тестирование и Валидация

Был разработан дополнительный функционал и пройден полный цикл тестирования:

1.  **Сборка:** проект успешно скомпилирован `cmake --build build --config Release --target ddos_collector`.
2.  **Запуск:** `.\ddos_collector.exe -i "Loopback" --pcap-dir pcap_dumps`
3.  **Симмуляция атаки:** с помощью скрипта `ddos_test.py` запущен 5-секундный UDP flood объёмом ~600,000 пакетов.
4.  **Проверка дампов:**
    - Использована библиотека `scapy`.
    - Подтверждено, что в `benign.pcap` сохранилось ~6,100 нормальных пакетов до начала атаки.
    - Подтверждено, что сотни тысяч пакетов успешно слились в `attack.pcap` (размер файла: ~370 MB).

Система работает стабильно, объемы ОЗУ находятся в пределах нормы, маркировка работает корректно.
