# Архитектурный Анализ и Идеи по Улучшению Проекта (Roadmap)

Проанализировав текущую архитектуру вашего DDoS-детектора (C++ Core + Qt GUI + PostgreSQL + ONNX), я подготовил список наиболее перспективных улучшений, которые выведут систему на уровень Enterprise и сделают дипломную работу "неубиваемой". 

Текущий софт отлично справляется со сбором статистики в реальном времени (доказано вашими локальными тестами на 130 000 пакетов в секунду), но есть фундаментальные узкие места, которые можно улучшить.

---

## 1. Активное подавление (Active Mitigation) через Windows Firewall API
**Проблема:**
Сейчас кнопка `Auto-Block Top-IP (BPF)` добавляет правило BPF фильтра прямо в Npcap. Это заставляет библиотеку "не замечать" трафик, **однако пакеты всё еще доходят до операционной системы**, нагружая сетевой стек Windows!
**Решение:**
Написать драйвер/интеграцию с `Windows Filtering Platform (WFP)` или использовать `netsh advfirewall` (через вызовы `CreateProcess`) для **реального** блокирования IP-адресов злоумышленников на уровне брандмауэра ОС. Это превратит ваш "Монитор" в полноценную систему IPS (Intrusion Prevention System).

## 2. Анализ с отслеживанием сессий (Per-Flow Tracking)
**Проблема:**
Сейчас движок `FeatureExtractor` агрегирует **весь** проходящий трафик (Single Window Aggregator) в одну метрику за секунду. То есть он вычисляет вероятность того, что вектор "весь сервер сейчас под атакой".
**Решение:**
Внедрить настоящую 5-Tuple хэш-таблицу (Src IP, Dst IP, Src Port, Dst Port, Protocol), чтобы накапливать `flow_duration`, `fwd_packets` и другие фичи для **каждого отдельного TCP/UDP соединения**. 
*Почему это круто для диплома:* Это позволит ML-модели находить точечные L7-атаки (например, Slowloris или HTTP Flood) среди гигабайтов легитимного трафика.

## 3. Защита от истощения БД при Volumetric Flood (Throttling)
**Проблема:**
Ваша система успешно обрабатывает 130k PPS, но если настроена запись каждого события в базу (`DatabaseManager::flushLoop`), во время DDoS атаки база будет расти со скоростью гигабайт в час, а потоки ввода-вывода (I/O) забьются, вызывая зависание Qt-интерфейса.
**Решение:**
Реализовать механизм **Adaptive Aggregation** или **Throttling**. Если модель уверена в DDoS на 99% уже 3 секунды подряд, новые пакеты из этой атаки не логировать индивидуально, а увеличивать счетчики единого объекта `AttackEvent` в памяти, и скидывать его в БД только по завершению атаки.

## 4. Генерация PDF-отчета об инцидентах (Reporting)
**Суть фичи:**
Для любой Enterprise-системы важна отчетность. Можно добавить кнопку **"Сформировать Security Report"**, которая через `QPrinter` и `QTextDocument` сгенерирует красивый PDF-документ с графиками (Heatmap, круговые диаграммы), описанием DDoS атаки, временем, Top-IPv4 векторами и вердиктом ONNX модели.
*Преимущество:* Эту PDF можно будет прямо распечатать и приложить в качества Приложения (Приложение А, Б) к Вашей дипломной работе.

## 5. Добавление умных метрик (TCP Flags, Entropy, Application Layer)
Сейчас парсер `FeatureExtractor` собирает объем и длины (Fwd Packets, Bwd Packets). Для более сложных ML моделей можно добавить умный парсинг флагов из `TcpLayer`:
- Количество `SYN`, `ACK`, `FIN`, `RST`
- Энтропия payload-а (для выявления рандомизированного мусорного трафика, UDP-флуда).
