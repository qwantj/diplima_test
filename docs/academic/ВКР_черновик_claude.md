# ВЫПУСКНАЯ КВАЛИФИКАЦИОННАЯ РАБОТА

## «Разработка программного комплекса обнаружения атак типа «отказ в обслуживании»»

## АННОТАЦИЯ

Объектом разработки является программный комплекс обнаружения DoS/DDoS-атак на базе C++17 и Qt6. Система включает модули захвата трафика и графического мониторинга. Для классификации 16 признаков сетевых потоков применяются модели XGBoost, Random Forest и MLP, интегрированные через ONNX Runtime с поддержкой «горячей замены». Реализована асинхронная запись в PostgreSQL и автоматическая блокировка угроз через Windows Firewall. Точность F1-score на датасете CIC-DDoS2019 составила 0,9997. Работа содержит 4 раздела, [N] страниц, 19 рисунков, 32 таблицы и 6 приложений.

**Abstract.** This thesis develops a C++17 and Qt6-based software complex for DoS/DDoS attack detection. The system features traffic analysis and GUI monitoring modules. Models (XGBoost, Random Forest, MLP) are integrated via ONNX Runtime with hot-swap support to classify 16 network flow features. The implementation includes asynchronous PostgreSQL storage and automated Windows Firewall mitigation. An F1-score of 0.9997 was achieved on the CIC-DDoS2019 dataset. The work comprises 4 sections, [N] pages, 19 figures, 32 tables, and 6 appendices.

## ОПРЕДЕЛЕНИЯ

1) **DoS-атака** (Denial of Service) — атака, целью которой является нарушение нормального функционирования сетевого сервиса путём исчерпания его ресурсов [7].
2) **DDoS-атака** (Distributed Denial of Service) — распределённая DoS-атака, осуществляемая одновременно с множества источников, как правило объединённых в ботнет [7].
3) **Пакет** — единица данных сетевого протокола, содержащая заголовок (служебную информацию) и полезную нагрузку [1].
4) **Поток (flow)** — совокупность пакетов, имеющих одинаковые значения 5-tuple: (src\_ip, dst\_ip, src\_port, dst\_port, protocol — IP-адрес источника, IP-адрес назначения, порт источника, порт назначения, протокол) [1].
5) **Признак трафика (feature)** — числовая характеристика пакета или потока, используемая моделью машинного или глубокого обучения для классификации [18].
6) **Классификатор** — алгоритм машинного и глубокого обучения, относящий объект к одному из заданных классов («атака» или «норма») [18].
7) **Межсетевой экран (firewall)** — программное или аппаратное средство фильтрации сетевого трафика по заданным правилам [7].
8) **Правило блокировки** — набор условий (IP-адрес, порт, протокол), при выполнении которых трафик отбрасывается [7].
9) **SRS (Software Requirements Specification)** — спецификация требований к программному обеспечению; документ, фиксирующий функциональные и нефункциональные требования к системе (стандарт IEEE 830) [19].
10) **Функциональное требование (FR)** — требование, описывающее конкретное действие или поведение системы [19].
11) **Нефункциональное требование (NFR)** — требование, описывающее характеристику работы системы: производительность, надёжность, безопасность, совместимость [19].
12) **Критерий приёмки (acceptance criterion)** — измеримое условие, при выполнении которого требование считается реализованным корректно [6].
13) **ONNX Runtime** — кроссплатформенная среда выполнения моделей машинного и глубокого обучения в формате ONNX (Open Neural Network Exchange) [25].
14) **Окно агрегации (inference window)** — временной интервал (1 секунда в данном комплексе), за который собираются пакеты и вычисляются признаки перед вызовом классификатора [34].
15) **РСУБД** (RDBMS) — реляционная система управления базами данных; программное обеспечение для управления данными в виде связанных таблиц [36].
16) **libpqxx** — официальный клиентский API для языка C++, предоставляющий доступ к возможностям СУБД PostgreSQL [38].
17) **Точность** (Precision) — доля истинно положительных срабатываний классификатора относительно всех объектов, отнесённых к положительному классу [18].
18) **Полнота** (Recall) — доля истинно положительных срабатываний классификатора относительно всех объектов, фактически принадлежащих к положительному классу [18].
19) **F1-мера** (F1-score) — гармоническое среднее точности и полноты, используемое для комплексной оценки качества классификации [18].
20) **ROC AUC** (Area Under the Receiver Operating Characteristic Curve) — площадь под кривой рабочих характеристик приёмника, количественный показатель качества классификатора [18].
21) **Многослойный перцептрон** (MLP) — архитектура нейронной сети прямого распространения, состоящая из входного, скрытых и выходного слоёв [28].
22) **XGBoost** (Extreme Gradient Boosting) — масштабируемая реализация алгоритма градиентного бустинга над решающими деревьями [12].
23) **Случайный лес** (Random Forest) — ансамблевый метод машинного обучения, основанный на построении множества независимых решающих деревьев [21].
24) **IoT** (Internet of Things) — сеть физических объектов («вещей»), оснащенных встроенными технологиями для взаимодействия друг с другом и с внешней средой [14].
25) **Смешанный режим** (promiscuous mode) — режим работы сетевого адаптера, позволяющий перехватывать все пакеты в сегменте сети, независимо от адреса назначения [1].
26) **Z-score нормализация** (стандартизация) — метод преобразования данных, при котором признаки центрируются относительно среднего значения и масштабируются по стандартному отклонению [18].
27) **Инференс** (inference) — этап эксплуатации модели машинного обучения, на котором она выполняет предсказания для входных данных [25].
28) **Матрица ошибок** (confusion matrix) — инструмент оценки качества классификации, представляющий собой таблицу распределения предсказаний модели по классам [18].
29) **Lock-free очередь** — потокобезопасная структура данных, реализующая алгоритм без блокировок, что исключает взаимные блокировки потоков и повышает производительность [24].
30) **C&C-сервер** (Command and Control) — центральный сервер, используемый злоумышленником для удалённого управления ботнетом и координации распределённых атак [14].
31) **1-SVM** (One-Class Support Vector Machine) — алгоритм обучения без учителя, используемый для обнаружения аномалий путём построения границы вокруг области нормальных данных [23].
32) **DPDK** (Data Plane Development Kit) — набор библиотек и драйверов для быстрой обработки сетевых пакетов путём обхода стека протоколов ядра операционной системы [1].
33) **OISF** (Open Information Security Foundation) — некоммерческая организация, занимающаяся разработкой и поддержкой открытых технологий обеспечения информационной безопасности, включая IDS Suricata [33].
34) **SIEM** (Security Information and Event Management) — класс программных продуктов, предназначенных для сбора, анализа и управления событиями безопасности в реальном времени [7].
35) **SAE** (Stacked Auto-Encoder) — архитектура глубокого обучения, состоящая из нескольких слоёв автоэнкодеров, обучаемая для извлечения иерархических признаков и подавления шума в данных [23].
36) **BPF** (Berkeley Packet Filter) — механизм в ядре операционной системы, позволяющий эффективно фильтровать и перехватывать пакеты на канальном уровне без лишнего копирования данных [1].
37) **ACID** (Atomicity, Consistency, Isolation, Durability) — набор свойств транзакций в СУБД, гарантирующих надёжность выполнения операций с данными даже в условиях аппаратных сбоев [36].
38) **Машинное обучение** (Machine Learning) — подобласть искусственного интеллекта, изучающая методы построения алгоритмов, способных обучаться на данных и делать предсказания [18].
39) **Глубокое обучение** (Deep Learning) — совокупность методов машинного обучения, основанных на использовании многослойных нейронных сетей для автоматического извлечения признаков из данных [28].
40) **Система обнаружения вторжений** (IDS) — программное или аппаратное средство, предназначенное для выявления фактов несанкционированного доступа в компьютерную систему или сеть [7].
41) **Система предотвращения вторжений** (IPS) — расширение IDS, способное не только обнаруживать, но и автоматически блокировать подозрительную активность в реальном времени [7].
42) **LSTM** (Long Short-Term Memory) — тип рекуррентной нейронной сети, способный обучаться долгосрочным зависимостям и эффективно обрабатывать временны́е последовательности [28].
43) **CNN** (Convolutional Neural Network) — архитектура нейронных сетей, использующая операцию свёртки для выявления локальных паттернов и иерархических структур в данных [28].
44) **SVM** (Support Vector Machine) — метод обучения с учителем, который находит оптимальную разделяющую гиперплоскость между классами в многомерном пространстве признаков [18].
45) **MPMC-очередь** (Multi-Producer Multi-Consumer) — структура данных, позволяющая нескольким потокам одновременно добавлять и извлекать элементы без нарушения целостности данных [24].
46) **Конструирование признаков** (Feature Engineering) — процесс преобразования сырых данных в информативные признаки, которые позволяют повысить точность моделей машинного обучения [18].

## ВВЕДЕНИЕ

**Актуальность.** В условиях глобальной цифровизации и стремительного расширения сетевой связности кибератаки типа «отказ в обслуживании» (DoS/DDoS) являются одной из наиболее распространённых и опасных угроз информационной безопасности. Ландшафт киберугроз в период 2024–2025 гг. демонстрирует беспрецедентный рост как мощности, так и интенсивности атак. Согласно аналитическим данным Cloudflare, в третьем квартале 2025 года число атак возросло на 40%, а в четвёртом квартале был зафиксирован рекордный пик мощности атаки в 31,4 Тбит/с при интенсивности 9,0 млрд пакетов в секунду [14, 15]. ### 1.1. Основные термины и классификация DoS/DDoS-атак

DoS-атаки (Denial of Service — отказ в обслуживании) представляют собой класс кибератак, целью которых является нарушение доступности сетевого сервиса для легитимных пользователей путём преднамеренного исчерпания двух видов ресурсов: пропускной способности каналов и вычислительных мощностей [21]. В отличие от DoS, атаки типа DDoS (Distributed Denial of Service) осуществляются одновременно с большого числа источников — как правило, с заражённых машин, входящих в состав ботнета [9].

### 1.2. Описание предметной области и ландшафт сетевых киберугроз

**Анатомия SYN Flood.** Протокол TCP предусматривает трёхстороннее рукопожатие для установки соединения: клиент отправляет SYN, сервер отвечает SYN-ACK, клиент завершает подтверждением ACK. При SYN Flood атакующий посылает огромное количество SYN-пакетов с подделанными (spoofed) IP-адресами источника. Сервер выделяет ресурсы (запись в таблице соединений) и отправляет SYN-ACK, но никогда не получает ACK. Таблица half-open соединений переполняется, и легитимные запросы отвергаются [21].

**Анатомия UDP Flood.** UDP — протокол без установки соединения (stateless). Атакующий генерирует огромное количество UDP-пакетов на случайные порты жертвы. Возможно усиление (amplification) через серверы DNS, NTP, SSDP: короткий запрос с подделанным IP генерирует ответ, многократно превосходящий его по объёму [11].

**Анатомия ICMP Flood.** Атакующий генерирует непрерывный поток ICMP Echo Request (ping) пакетов. Атака Smurf использует широковещательную рассылку ICMP Echo Request на адрес сети с подделанным адресом источника (жертвы): все хосты сети отвечают одновременно, создавая эффект усиления, пропорциональный числу хостов [7].

**Атаки прикладного уровня.** HTTP GET/POST Flood имитирует легитимные запросы к веб-серверу, из-за чего сигнатурные системы с трудом их отличают. Особую сложность представляет атака **Slowloris**: злоумышленник искусственно затягивает время TCP-сессий путём крайне медленной отправки фрагментированных HTTP-запросов, исчерпывая пул доступных соединений веб-сервера без генерации больших объёмов трафика. Стандартные методы ограничения скорости часто не способны отличить подобную активность от работы легитимных клиентов с медленным соединением [17].

**Ботнеты как инфраструктура DDoS.** Распределённые атаки осуществляются через ботнеты — сети из тысяч заражённых устройств (ботов), управляемых через командный сервер C&C. Распространение уязвимых IoT-устройств (умные телевизоры, маршрутизаторы) привело к появлению ботнетов нового поколения, таких как Mirai [10] и Aisuru-Kimwolf [14], способных генерировать гиперволюметрические атаки. Это делает фильтрацию по IP-адресу источника крайне неэффективной.

### 1.3. Обзор существующих решений: преимущества и недостатки

Проведён комплексный сравнительный анализ разрабатываемого программного комплекса с тремя классами существующих решений.

#### 1.3.1. Традиционные сигнатурные системы (Snort, Suricata, Zeek)

**Snort** — система обнаружения вторжений с открытым исходным кодом, разработанная Мартином Рошем в 1998 году и ныне поддерживаемая компанией Cisco. Работает по принципу сигнатурного анализа: каждый пакет сверяется с набором правил на языке Snort Rule Language. Достоинством является зрелость и обширная база правил. Недостатком — полная зависимость от актуальности сигнатур: новые виды атак («нулевого дня») остаются незамеченными [33, 35].

**Suricata** — многопоточная IDS/IPS с открытым исходным кодом (OISF). Поддерживает многопоточную обработку пакетов, анализ протоколов прикладного уровня и режим IPS (через nfqueue). Содержит зачатки поведенческого анализа (threshold, flow keywords). Однако по-прежнему основывается на сигнатурах [33, 35].

Ключевое отличие разрабатываемого решения от Snort/Suricata: переход от жёсткого сопоставления байтовых шаблонов к интеллектуальному выявлению аномалий на базе ML/DL, что обеспечивает устойчивость к атакам «нулевого дня» [21, 23]. Также исключается деградация производительности, неизбежная в сигнатурных системах при разрастании баз правил [22].

#### 1.3.2. Коммерческие облачные сервисы (Cloudflare, Arbor Networks)

Коммерческие средства защиты от DDoS используют глобальные распределённые центры очистки трафика (Scrubbing Centers) [13, 15]. Их преимущество — способность абсорбировать гиперволюметрические атаки мощностью сотни гигабит и терабит в секунду. Недостаток разрабатываемого решения в этом сравнении: локальный комплекс физически не способен противостоять таким атакам, превышающим пропускную способность каналообразующего оборудования провайдера.

Преимущество разрабатываемого решения: полный контроль над конфиденциальностью данных, независимость от сторонних провайдеров (отсутствие Vendor Lock-in), что критически важно для защиты объектов критической информационной инфраструктуры (КИИ). Прозрачность алгоритмов принятия решений («белый ящик» против «чёрного ящика» облачных сервисов).

#### 1.3.3. Локальные программные аналоги (FastNetMon, PRTG Network Monitor)

**FastNetMon** — локальная система обнаружения DDoS-атак на основе анализа NetFlow/sFlow. Лишена нативной ML/DL-классификации; требует развёртывания сторонних стеков визуализации (Grafana, InfluxDB) [16]. **PRTG Network Monitor** — система мониторинга сети, не ориентированная на обнаружение атак в реальном времени [27].

Ключевое отличие разрабатываемого комплекса: нативная интеграция ONNX Runtime непосредственно в конвейер C++ обработки пакетов [25, 26] и встроенный Qt6-монитор «из коробки» без необходимости развёртывания сторонних стеков [22]. ### 1.4. Теоретические основы машинного и глубокого обучения для обнаружения атак

#### 1.4.1. Обзор методов машинного и глубокого обучения для классификации трафика

Выбор конкретного ML/DL-алгоритма определяет баланс между точностью детекции и задержкой инференса [21]. В программном комплексе реализована поддержка трёх архитектур классификаторов, обеспечивающая возможность сравнительного анализа.

**1.4.1.1. Алгоритм «Случайный лес» (Random Forest)**

Random Forest (RF) представляет собой ансамбль решающих деревьев, обучаемых независимо на различных подвыборках данных (bagging) и случайных подмножествах признаков. Финальное решение принимается путём голосования. Логарифмическая сложность инференса позволяет обрабатывать пакеты на скоростях, близких к линейным, что обеспечивает первичную фильтрацию объёмных атак [21]. Преимущества RF в задаче обнаружения DoS-атак: устойчивость к переобучению, способность обрабатывать нелинейные зависимости в признаках трафика и внутренняя оценка значимости признаков (feature importance).

**1.4.1.2. Многослойный перцептрон (MLP)**

Многослойный перцептрон — классическая архитектура нейронной сети прямого распространения. В данной работе применяется архитектура с двумя скрытыми слоями (64 и 32 нейрона), функцией активации ReLU и оптимизатором Adam с механизмом `early_stopping` для предотвращения переобучения [28]. MLP способен выявлять нелинейные зависимости в пространстве признаков, недоступные деревьям решений.

**1.4.1.3. Архитектуры глубокого обучения (CNN-LSTM)**

Для выявления сложных паттернов ботнетов типа Mirai [10] применяются нейронные сети специальной архитектуры: CNN-слои выполняют компрессию и извлечение пространственных признаков (размеры пакетов, флаги) с минимальной задержкой, LSTM-блоки обрабатывают эти признаки для выявления временных аномалий. Синергия **CNN-LSTM** позволяет достичь высокой точности детектирования атак, растянутых во времени, таких как Slowloris [17, 28].

**1.4.1.4. Автоэнкодеры (SAE) и атаки «нулевого дня»**

В условиях дефицита размеченных данных применяются методы обучения без учителя. Стековые автоэнкодеры (SAE) обучаются восстанавливать профиль нормального поведения сети. Данные, не соответствующие этому профилю (атаки Zero-day), генерируют высокую ошибку реконструкции. Гибрид SAE-1SVM обеспечивает точность свыше 99% при обнаружении атак «нулевого дня» [23].

**Обоснование выбора.** В разрабатываемом комплексе основной классификатор — алгоритм градиентного бустинга (XGBoost), экспортированный в формат ONNX, как обеспечивающий наилучший баланс: задержка инференса <1 мс, F1-score = 0,9997 на датасете CIC-DDoS2019 [34]. RF и MLP доступны для сравнения и горячей замены без перекомпиляции.

#### 1.4.3. Основной алгоритм классификации: экстремальный градиентный бустинг (XGBoost)

В качестве базового классификатора в разрабатываемом программном комплексе выбран алгоритм экстремального градиентного бустинга (XGBoost), обеспечивающий наилучшие показатели точности и скорости инференса при работе с табличными данными сетевых потоков. XGBoost реализует аддитивную стратегию обучения, при которой каждое последующее дерево исправляет ошибки предыдущих, минимизируя регуляризованную целевую функцию:

$$ L(\phi) = \sum_i l(\hat{y}_i, y_i) + \sum_k \Omega(f_k) \eqno(1.1) $$

где $L(\phi)$ — общая целевая функция;
$l(\hat{y}_i, y_i)$ — функция потерь, оценивающая разность между предсказанием $\hat{y}_i$ и истинной меткой $y_i$;
$\Omega(f_k) = \gamma T + \frac{1}{2}\lambda\|w\|^2$ — регуляризационный член, предотвращающий переобучение путём штрафования сложности деревьев (количества листьев $T$ и весов $w$).

Особенностью XGBoost, критически важной для обнаружения DDoS-атак в реальном времени, является использование разложения Тейлора второго порядка для аппроксимации функции потерь:

$$ L^{(t)} \approx \sum_{i=1}^n [l(y_i, \hat{y}_i^{(t-1)}) + g_i f_t(x_i) + \frac{1}{2} h_i f_t^2(x_i)] + \Omega(f_t) \eqno(1.2) $$

где $g_i = \partial_{\hat{y}^{(t-1)}} l(y_i, \hat{y}_i^{(t-1)})$ — градиент (первая производная) функции потерь;
$h_i = \partial^2_{\hat{y}^{(t-1)}} l(y_i, \hat{y}_i^{(t-1)})$ — гессиан (вторая производная) функции потерь.

Аналитическое решение для нахождения оптимального веса в листе $j$ дерева имеет вид:

$$ w_j^* = -\frac{\sum_{i \in I_j} g_i}{\sum_{i \in I_j} h_i + \lambda} \eqno(1.3) $$

Такой подход позволяет алгоритму эффективно обрабатывать разреженные данные сетевого трафика и обеспечивать высокую скорость принятия решений (инференс < 1 мс), что является ключевым преимуществом при отражении интенсивных атак. Визуализация процесса последовательного уточнения предсказаний через построение ансамбля деревьев представлена на рис. 1.2.

```mermaid

    D1[Дерево 1] -- "Ошибка (L1)" --> D2[Дерево 2]
    D2 -- "Ошибка (L2)" --> D3[Дерево 3]

    Input[Признаки трафика] --> D1
    Input --> D2
    Input --> D3
    Input --> Dn
    D1 & D2 & D3 & Dn --> Sum(Σ Сумма весов листьев)

```

#### 1.4.4. Ансамблевый метод классификации на основе случайного леса (Random Forest)

Алгоритм «Случайный лес» используется в системе как альтернативный классификатор, обладающий высокой устойчивостью к выбросам в признаках трафика. Метод основан на построении ансамбля из $N$ независимых решающих деревьев с использованием стратегий Bagging (Bootstrap aggregating) и Random Subspace Method. Итоговое предсказание $\hat{y}$ определяется путём мажоритарного голосования всех деревьев:

$$ \hat{y} = \text{argmax}_{c \in C} \sum_{n=1}^N I(f_n(x) = c) \eqno(1.4) $$

где $f_n(x)$ — предсказание $n$-го дерева;
$I(\cdot)$ — индикаторная функция;
$C$ — множество классов {0 — норма, 1 — атака}.

Для разделения узлов в каждом дереве используется критерий информативности — индекс Джини (Gini Impurity), минимизирующий вероятность ошибочной классификации:

$$ Gini(p) = \sum_{k=1}^K p_k(1-p_k) = 1 - \sum_{k=1}^K p_k^2 \eqno(1.5) $$

где $p_k$ — доля объектов класса $k$ в рассматриваемом узле. Структурная схема организации ансамбля деревьев и механизма принятия коллективного решения методом голосования приведена на рис. 1.3.

```mermaid

    Data[Обучающая выборка] --> B1[Bootstrap 1]
    Data --> B2[Bootstrap 2]
    Data --> Bn[Bootstrap N]
    B1 --> T1[Дерево 1]
    B2 --> T2[Дерево 2]
    Bn --> Tn[Дерево N]
    T1 & T2 & Tn --> Vote{Мажоритарное голосование}

```

#### 1.4.5. Архитектура классификатора на базе многослойного перцептрона (MLP)

Для выявления сложных нелинейных зависимостей между признаками, такими как энтропия полезной нагрузки и динамика TCP-окон, реализована нейросетевая модель MLP. Архитектура сети представляет собой полносвязный граф, состоящий из входного слоя, двух скрытых слоёв и выходного слоя с сигмоидальной функцией активации.

Значение активации нейрона $j$ в слое $l$ вычисляется как:

$$ a_j^{(l)} = \sigma \left( \sum_k w_{jk}^{(l)} a_k^{(l-1)} + b_j^{(l)} \right) \eqno(1.6) $$

где $w$ — веса связей, $b$ — смещение (bias), $\sigma$ — функция активации. В скрытых слоях $\sigma$ принимает значение ReLU ($\sigma(x) = \max(0, x)$), в выходном нейроне — Sigmoid ($\sigma(x) = 1/(1+e^{-x})$). Графическое представление топологии нейронной сети с выделением входного, скрытых и выходного слоёв показано на рис. 1.4.

```mermaid
graph LR
    subgraph Input[Входной слой]
        X1((f1))
        X2((f2))
        Xn((fn))
    end
    subgraph Hidden[Скрытые слои ReLU]
        H1((h1))
        H2((h2))
        H3((h3))
    end
    subgraph Output[Выходной слой Sigmoid]
        Y((Label))
    end
    X1 & X2 & Xn --> H1 & H2 & H3
    H1 & H2 & H3 --> Y
```
<p align="center">Рис. 1.4. Топология многослойного перцептрона</p>

Обучение модели выполняется методом обратного распространения ошибки (Backpropagation) путём минимизации функции кросс-энтропии:

$$ J(w,b) = -\frac{1}{m} \sum_{i=1}^m [y_i \ln(\hat{y}_i) + (1-y_i) \ln(1-\hat{y}_i)] \eqno(1.7) $$

Применение MLP позволяет системе адаптироваться к тонким изменениям в структуре трафика, характерным для атак прикладного уровня, которые могут имитировать поведение легитимных клиентов.

#### 1.4.6. Метрики оценки качества классификатора

Для оценки эффективности классификатора используются стандартные метрики бинарной классификации. Введём обозначения: $TP$ (True Positive) — правильно обнаруженные атаки, $FP$ (False Positive) — ложные тревоги, $FN$ (False Negative) — пропущенные атаки, $TN$ (True Negative) — правильно классифицированный нормальный трафик.

- **Точность (Precision):** доля реальных атак среди всех срабатываний системы:
$$ P = \frac{TP}{TP + FP} \eqno(1.8) $$

- **Полнота (Recall):** доля обнаруженных атак от их фактического количества:
$$ R = \frac{TP}{TP + FN} \eqno(1.9) $$

- **F1-мера:** гармоническое среднее точности и полноты, обеспечивающее комплексную оценку качества:
$$ F_1 = \frac{2 \cdot P \cdot R}{P + R} \eqno(1.10) $$

- **ROC AUC:** площадь под кривой ROC, характеризующая качество ранжирования классификатора. Для оценки качества работы классификатора используется матрица ошибок, структура которой приведена в табл. 1.4.

| — | Предсказано: атака | Предсказано: норма |
|---|---|---|
| **Реально: атака** | TP (верно обнаружена атака) | FN (пропуск атаки) |
| **Реально: норма** | FP (ложная тревога) | TN (верно определена норма) |

Для систем безопасности критически важно минимизировать $FN$ (пропущенные атаки), поэтому целевым показателем является $Recall \geq 0,88$. F1-мера балансирует между точностью и полнотой.

#### 1.4.7. Нормализация признаков

Перед подачей вектора признаков в классификатор каждый признак нормализуется методом стандартизации (z-score) по формуле:

$$ x'_i = \frac{x_i - \mu_i}{\sigma_i} \eqno(1.11) $$

где $x'_i$ – нормализованное значение $i$-го признака;
$x_i$ – исходное значение признака;
$\mu_i$ – среднее значение $i$-го признака на обучающей выборке;
$\sigma_i$ – стандартное отклонение $i$-го признака.

Параметры μ и σ вычисляются при обучении модели и сохраняются в JSON-файл конфигурации масштабировщика (`scaler_params.JSON`). При инициализации комплекса эти параметры загружаются в структуру `FeatureExtractor::ScalerParams`. Для признаков с тяжёлым правым хвостом (например, byte rate) дополнительно может применяться логарифмическое преобразование log1p перед стандартизацией (`useLog1p = true`).

Основной классификатор на базе XGBoost позволяет достичь минимальной задержки инференса при рекордной точности. Применение ONNX Runtime обеспечивает кроссплатформенную среду выполнения без Python-зависимостей. Выделенный набор из 16 признаков охватывает поведение TCP-флагов, интенсивность потоков и их временну́ю динамику, что достаточно для эффективного разделения DoS/DDoS-атак и нормального трафика.

**2.2.2. Класс FeatureExtractor**

Класс `FeatureExtractor` выполняет агрегацию сырых пакетов в логические сетевые потоки и вычисляет вектор признаков для классификатора. Основная сложность реализации данного модуля заключается в необходимости отслеживания двунаправленных потоков (bidirectional flows) в реальном времени.

**2.2.3. Класс ModelInferencer**

Класс `ModelInferencer` является инкапсуляцией среды выполнения ONNX Runtime. Его основная задача — загрузка предварительно обученных ML/DL-моделей из файлов формата `.ONNX` и выполнение предиктивного анализа на векторе признаков.

Реализация класса поддерживает механизм «горячей замены» (hot-swap) модели без остановки процесса захвата трафика. Это достигается за счёт использования примитива синхронизации `std::shared_mutex`: процесс инференса захватывает «разделяемую» блокировку (read lock), а процесс обновления модели — «эксклюзивную» (write lock). Такой подход гарантирует потокобезопасность и непрерывность мониторинга. На рис. 2.5 приведена структурная схема интеграции ONNX Runtime, а в табл. 2.4 описан программный интерфейс класса.

**2.2.6. Класс FirewallManager**

**2.2.7. Класс PcapDumper**

Класс `PcapDumper` предназначен для записи захваченного сетевого трафика в файлы формата `.pcap`. Это необходимо для последующего офлайн-анализа инцидентов, расследования причин атак и формирования доказательной базы. Модуль инкапсулирует работу с классом `pcpp::PcapFileWriterDevice` и обеспечивает потокобезопасную запись пакетов в фоновом режиме.

Активация записи происходит автоматически при обнаружении атаки, если в конфигурации взведен флаг `dump_enabled`. Файлы именуются по шаблону `attack_<timestamp>.pcap` и сохраняются в директорию, указанную в параметре `pcap_dump_dir`. По завершении инцидента файл закрывается, что гарантирует целостность данных.

### 2.3. Проектирование структуры базы данных

Для обеспечения возможности последующего анализа инцидентов, формирования статистической отчетности и визуализации состояния системы в реальном времени спроектирована специализированная структура базы данных. В качестве хранилища выбрана СУБД PostgreSQL 15+, обеспечивающая необходимую производительность при интенсивной записи потоковых данных и обладающая развитыми средствами асинхронного взаимодействия.

Архитектура базы данных построена по иерархическому принципу, что позволяет эффективно структурировать информацию разной степени детализации: от метаданных сессий мониторинга до векторов признаков каждого отдельного потока. Ключевой особенностью проектирования является разделение «горячих» данных (оперативная статистика) и «холодных» данных (завершенные инциденты), что минимизирует накладные расходы на индексацию при высокой нагрузке.

Взаимодействие с сервером осуществляется через объектно-ориентированную библиотеку libpqxx [30]. Для реализации нефункционального требования NFR-001 (производительность) при проектировании БД были применены следующие технические решения:
1) **Асинхронная пакетная запись:** использование паттерна «Producer-Consumer» с lock-free очередью позволяет изолировать основной поток обработки трафика от задержек дисковой подсистемы БД.
2) **Механизм COPY (потоковая вставка):** вместо выполнения множественных одиночных запросов `INSERT`, система использует интерфейс `stream_to` [30], инициирующий команду `COPY` в PostgreSQL. Это сокращает накладные расходы на парсинг SQL-запросов и транзакционную нагрузку в 5–7 раз.
3) **Локальная отказоустойчивость:** внедрен промежуточный класс `FileBuffer`, выполняющий роль «теневой копии» данных при временной недоступности сетевого соединения с сервером БД.

### 2.4. Разработка схемы алгоритмов

Логика функционирования программного комплекса декомпозирована на три ключевых алгоритма, описывающих жизненный цикл обработки данных, процесс математической подготовки признаков и механизм активного противодействия угрозам.

#### 2.4.1. Алгоритм главного цикла обработки (инференс-цикл)

Основной цикл обработки, инкапсулированный в классе `DetectionEngine`, реализует стратегию непрерывного мониторинга. Алгоритм спроектирован так, чтобы минимизировать время прохождения данных от сетевого интерфейса до классификатора, обеспечивая при этом защиту от ложных срабатываний на незначительном сетевом фоне.

Ключевой особенностью является разделение на «быстрый путь» (захват пакетов в очередь) и «аналитический путь» (извлечение признаков и инференс), работающих в параллельных потоках. На рис. 2.10 представлена схема алгоритма, включающая этап инициализации, фильтрацию фонового шума и логику принятия решения о блокировке.

```mermaid
flowchart TD
    Start(["Начало"]) --> Init["Инициализация подсистем\nи загрузка конфигурации"]
    Init --> Capture["Запуск асинхронного\nзахвата сетевого трафика"]
    Capture --> LoopStart["Начало нового окна агрегации (1 с)\nи сброс счетчиков"]
    LoopStart --> Dequeue{"В очереди есть\nнеобработанный пакет?"}
    Dequeue -- Да --> Process["Обработка пакета и\nобновление статистики потока"]
    Process --> TimeCheck{"Окно агрегации\nзавершено?"}
    TimeCheck -- Нет --> Dequeue
    TimeCheck -- Да --> Telemetry["Вычисление агрегированной\nтелеметрии (пак/с)"]
    Dequeue -- Нет --> TimeCheck
    Telemetry --> Noise{"Интенсивность\n< 50 пак/с?"}
    Noise -- Да --> Benign["Присвоение метки\nлегитимного трафика"]
    Noise -- Нет --> Features["Масштабирование и нормализация\nвектора признаков"]
    Features --> Infer["Классификация потока\nмоделью машинного обучения"]
    Infer --> Incident["Обновление текущего\nсостояния инцидента"]
    Incident --> Block{"Атака обнаружена\nи защита включена?"}
    Block -- Да --> Firewall["Блокировка IP-адреса источника\nна межсетевом экране"]
    Block -- Нет --> Callback
    Firewall --> Callback["Сохранение события в БД\nи отправка телеметрии"]
    Benign --> Callback
    Callback --> Running{"Продолжить\nработу?"}
    Running -- Да --> LoopStart
    Running -- Нет --> Stop(["Конец\n(очистка правил блокировки)"])
```

<p align="center">Рис. 2.10. Схема алгоритма главного цикла</p>

Алгоритм работы подсистемы `ddos_collector`:

1) Инициализация ConfigManager: загрузка config.JSON (путь к модели, параметры БД, имя интерфейса)
2) Инициализация ModelInferencer: загрузка ONNX-модели через `Ort::Session`
3) Загрузка параметров масштабировщика: JSON-файл → `FeatureExtractor::ScalerParams`
4) Подключение к PostgreSQL; создание таблиц (если не существуют); создание сессии
5) Инициализация TcpServer для потоковой передачи данных в `ddos_monitor`
6) Вызов `TrafficMonitor::startCapture()` → захват пакетов в lock-free очередь
7) Запуск потока инференса (`DetectionEngine::inferenceLoop()`):
   - Начало окна агрегации (1 секунда): `FeatureExtractor::reset()`
   - Цикл извлечения пакетов: `dequeuePacket()` → `processPacket()`
   - По истечении окна: `computeNormalizedFeatures()` → вектор признаков
   - Проверка порога шума (PPS < 50): если слишком тихо — метка «норма» без инференса
   - `ModelInferencer::predict()` → (метка, уверенность)
   - `updateIncidentState()`: отслеживание переходов атака↔норма
   - Если атака и mitigation включён: `FirewallManager::blockIp(topAttacker)`
   - Если атака завершилась: `FirewallManager::unblockAllIps()`
   - `resultCallback_()` → DatabaseManager (async enqueue) + TcpServer (live send)
8) При остановке: `stopCapture()`, `closeSession()`, очистка правил фаервола

#### 2.4.2. Алгоритм извлечения и нормализации признаков

Алгоритм извлечения признаков выполняет критически важную трансформацию сырых байтовых данных в структурированное признаковое описание сетевого поведения. Процесс основан на глубоком разборе заголовков пакетов уровней L2-L4 и агрегации статистики в рамках скользящего временного окна.

Применение агрегации по всем активным потокам позволяет системе детектировать не только точечные атаки на конкретный порт, но и распределенные воздействия (DDoS), характеризующиеся аномальным всплеском активности по всему интерфейсу. Завершающий этап алгоритма — математическая стандартизация — гарантирует, что входные данные для ML/DL-моделей будут находиться в сопоставимых диапазонах, что исключает доминирование признаков с большой размерностью (например, количество байт) над флагами. На рис. 2.11 детально показан процесс парсинга пакетов и пайплайн нормализации.

```mermaid
flowchart TD
    Start(["Начало обработки пакета"]) --> Parse["Синтаксический анализ пакета\n(извлечение заголовков L2-L4)"]
    Parse --> Tuple["Формирование идентификатора потока\n(5-tuple: IP и порты источника/назначения, протокол)"]
    Tuple --> FindFlow{"Поток существует\nв таблице?"}

    Proto -- TCP --> TCP_Flags["Обновление TCP-метрик\n(счетчики флагов, размер окна)"]
    Proto -- UDP --> UDP_End["Обновление UDP-метрик\n(полезная нагрузка)"]
    Proto -- ICMP --> ICMP_End["Обновление ICMP-метрик"]
```

```mermaid
flowchart TD
    Start(["Начало обработки окна"]) --> NF1["Формирование исходного вектора\nпризнаков для каждого потока"]
    NF1 --> NF2{"Требуется логарифмическое\nсглаживание?"}
    NF2 -- Да --> NF3["Масштабирование значений\nпризнаков (функция log1p)"]
```

<p align="center">Рис. 2.11. Схема алгоритма извлечения признаков</p>

1) Получить `PacketBuffer*` из очереди (содержит сырые байты и метку времени)
2) Распаковать через PcapPlusPlus: `Ethernet → IP → TCP/UDP/ICMP`
3) Извлечь 5-tuple: (src\_ip, dst\_ip, src\_port, dst\_port, proto)
4) Найти или создать запись `FlowStats` в `activeFlows_[FlowKey]`
5) Обновить счётчики потока: `fwdPackets++`, `fwdBytes += len`
6) Если TCP: обновить счётчики флагов (`synPackets`, `ackPackets`, `finPackets`, `rstPackets`, `pshPackets`, `urgPackets`); обновить `tcpWindowSizeTotal`
7) Обновить глобальные счётчики окна: `totalPackets_`, `tcpPackets_` / `udpPackets_` / `icmpPackets_`, `totalBytes_`
8) По окончании окна (`computeNormalizedFeatures()`):
   - Для каждого активного потока: вычислить вектор признаков из `FlowStats`
   - Применить z-score: `x' = (x - μᵢ) / σᵢ`
   - Если `useLog1p`: предварительно `x = log(1 + x)` перед нормализацией
   - Вернуть вектор `vector<double>` (один вектор для «тяжёлого» потока или батч)

#### 2.4.3. Алгоритм блокировки источника атаки

Алгоритм блокировки реализует реактивную часть системы защиты. Его основной задачей является оперативное внесение изменений в правила фильтрации операционной системы при обнаружении деструктивного воздействия, подтвержденного классификатором.

Логика алгоритма построена на концепции конечного автомата, отслеживающего переходы между состояниями «Норма» и «Атака». Это позволяет не только мгновенно блокировать нарушителя через системные вызовы Windows Firewall, но и автоматически восстанавливать доступ после прекращения атаки, исключая избыточную нагрузку на системный брандмауэр и предотвращая случайную блокировку легитимных пользователей на длительный срок. На рис. 2.12 изображена логика взаимодействия с брандмауэром и управление списком активных блокировок.

```mermaid
flowchart TD
    Start(["Начало: фиксация начала атаки"]) --> ExtractIPs["Извлечение списка наиболее\nактивных IP-адресов"]

    CheckMitigation -- Нет --> EndOnlyDB(["Конец: только регистрация\nсобытия в БД"])
    CheckMitigation -- Да --> GetTopIP["Выбор IP-адреса\nосновного источника атаки"]
```

```mermaid
    Start(["Начало: фиксация окончания атаки"]) --> UpdateState["Обновление статуса инцидента\nи вызов функции оповещения"]
```

<p align="center">Рис. 2.12. Схема алгоритма блокировки</p>

1) Получить список topTalkers из `DetectionResult` (top IP-источников по числу пакетов)
2) Проверить: не находится ли IP уже в `activeBlockedIps_` (мьютекс `g_firewallMutex`)
3) Если уже заблокирован — пропустить (идемпотентная операция)
4) Если не заблокирован:
   - Валидировать формат IP через `QHostAddress`
   - Сформировать команду: `netsh advfirewall firewall add rule name=DDoS_Block_<IP> dir=in action=block remoteip=<IP>`
   - Выполнить через `QProcess::startDetached()`
   - Добавить IP в `activeBlockedIps_`
5) При переходе «атака → норма» (в `DetectionEngine::updateIncidentState()`):
   - Вызвать `FirewallManager::unblockAllIps()`
   - Для каждого IP в `activeBlockedIps_`: выполнить `netsh advfirewall firewall delete rule name=DDoS_Block_<IP>`
   - Очистить `activeBlockedIps_`

### 2.6. Разработка пользовательского интерфейса

#### 2.6.1. Категории пользователей и выполняемые функции

Программный комплекс предназначен для эксплуатации в отделах обеспечения информационной безопасности предприятий. Основной категорией пользователей системы является **Администратор информационной безопасности (оператор ИБ)**. 

В соответствии с ролью администратора, система обеспечивает выполнение следующих функций:
1) **Конфигурирование системы**: выбор сетевого интерфейса, загрузка весов ML-модели, настройка параметров подключения к БД.
2) **Оперативный мониторинг**: наблюдение за интенсивностью трафика (PPS) и результатами классификации в реальном времени.

### 3.1. Захват пакетов и инициализация PcapPlusPlus

Захват сетевых пакетов реализован с использованием библиотеки PcapPlusPlus — высокоуровневой объектно-ориентированной обёртки над Npcap/libpcap. Выбор данной библиотеки обусловлен необходимостью обеспечения высокой производительности при сохранении типобезопасности и удобства управления ресурсами в языке C++.

В процессе инициализации подсистемы `ddos_collector` выполняется сканирование доступных сетевых интерфейсов. Ключевым требованием является поддержка драйвером режима «смешанного захвата» (promiscuous mode), что позволяет перехватывать все пакеты в сегменте сети, включая те, которые не адресованы непосредственно хосту мониторинга. Инициализация выполняется в методе `TrafficMonitor::startCapture()`. Для поиска устройства по имени или описанию реализована нечёткая двухэтапная стратегия поиска: сначала ищется точное совпадение по имени или IPv4-адресу, затем — по очищенному описанию с приоритетом физических устройств над виртуальными (Hyper-V, Microsoft NDIS):

Листинг 1 (фрагмент) — Инициализация устройства захвата
```cpp
  // 1. Поиск по точному совпадению имени или IP
  for (auto* dev : devList) {
    if (dev->getName() == interfaceName || 
        dev->getIPv4Address().toString() == interfaceName) {
      device_ = dev; break;
    }
  }

  // 2. Нечеткий поиск по описанию с приоритетом физических устройств
  if (!device_) {
    pcpp::PcapLiveDevice* candidate = nullptr;
    for (auto* dev : devList) {
      std::string desc = dev->getDesc();
      // (очистка и поиск подстроки)
      if (desc.find(interfaceName) != std::string::npos) {
        bool isVirtual = desc.find("virtual") != std::string::npos;
        if (!isVirtual) { device_ = dev; break; }
        else if (!candidate) candidate = dev;
      }
    }
    if (!device_) device_ = candidate;
  }
```

Полный исходный код метода `startCapture` и статического обработчика пакетов `onPacketArrived` приведён в приложении Б (листинг Б.1).

Для минимизации накладных расходов на выделение и освобождение памяти (heap allocation) в режиме реального времени в классе `TrafficMonitor` реализован паттерн «Пул объектов» (Object Pool) на базе структуры `PacketBuffer`. Пакеты, поступающие от драйвера, копируются в предварительно аллоцированные буферы и помещаются в lock-free очередь для последующего анализа. После обработки признаков буфер возвращается в пул для повторного использования. Максимальный размер очереди `MAX_QUEUE_SIZE = 500 000` пакетов обеспечивает буферизацию при кратковременных пиках нагрузки, предотвращая потерю данных при всплесках трафика (bursts).

### 3.2. Извлечение признаков и агрегация по потокам

Класс `FeatureExtractor` выполняет критически важную задачу преобразования сырого пакетного трафика в структурированный вектор признаков. В отличие от систем, анализирующих пакеты изолированно, данный комплекс использует потоковую агрегацию (flow aggregation).

Каждый пакет идентифицируется по уникальному 5-tuple: (IP-источника, IP-назначения, порт-источника, порт-назначения, протокол). Для обеспечения корректного анализа двунаправленных сессий (bidirectional flows) в классе реализована нормализация ключа `FlowKey`: адреса и порты сортируются лексикографически, что позволяет объединять пакеты прямого и обратного направлений в одну структуру `FlowStats`. Статистика накапливается в хеш-таблице `std::map` в рамках скользящего окна длительностью 1 секунда.

Листинг 2 — Структуры сетевого потока
```cpp
// Уникальный ключ потока (симметричный 5-tuple)
struct FlowKey {
  std::string ip1, ip2; // IP-адреса, отсортированные лексикографически
  uint16_t port1, port2;
  uint8_t proto;
  // ... операторы сравнения для std::map
};

// Статистические показатели потока за текущее окно агрегации
struct FlowStats {
  std::string initiatorIp;
  uint64_t fwdPackets = 0, bwdPackets = 0;
  uint64_t fwdBytes = 0, bwdBytes = 0;
  // ... счетчики флагов и энтропии
};
```

Математическая подготовка данных включает расчет 16 информативных признаков, включая интенсивность (PPS), распределение TCP-флагов и энтропию полезной нагрузки. Поскольку признаки имеют разную физическую природу и масштабы (например, количество пакетов измеряется единицами, а объём трафика — миллионами байт), обязательным этапом является стандартизация (z-score) по формуле (1.11), приведенной в подразделе 1.4.7. Это исключает доминирование признаков с большими абсолютными значениями при обучении и инференсе ML-модели. Дополнительно поддерживается log1p-преобразование для признаков с тяжёлым хвостом, что позволяет сгладить влияние экстремальных выбросов.

Листинг 3 (фрагмент) — Итерация по признакам при нормализации
```cpp
  // Стандартизация: (x - mean) / std
  for (size_t i = 0; i < raw.size(); ++i) {
    double val = raw[i];
    if (scaler_.useLog1p) val = std::log1p(std::max(0.0, val));
    normalized[i] = (val - scaler_.mean[i]) / 
                    (std::abs(scaler_.scale[i]) > 1e-10 ? scaler_.scale[i] : 1.0);
  }
```

### 3.3. ML/DL-инференс через ONNX Runtime

Для выполнения предиктивного анализа в режиме реального времени выбран кроссплатформенный движок ONNX Runtime (ORT). Использование формата ONNX (Open Neural Network Exchange) позволяет разделить этапы обучения (на языке Python с использованием XGBoost, Scikit-Learn или PyTorch) и этап эксплуатации (на языке C++). Это исключает необходимость развёртывания тяжеловесного Python-интерпретатора в производственной среде и обеспечивает высокую скорость инференса за счёт использования оптимизированных графов вычислений. Реализованная обёртка также поддерживает возможность задействования аппаратного ускорения (CUDA, DirectML) через соответствующие Execution Providers, что позволяет масштабировать комплекс на высоконагруженных узлах с графическими сопроцессорами.

Класс `ModelInferencer` инкапсулирует API ONNX Runtime и управляет жизненным циклом сессии (`Ort::Session`). Одной из ключевых особенностей реализации является поддержка «горячей замены» (hot-swap) модели без остановки процесса сбора данных. Для обеспечения потокобезопасности при одновременном выполнении анализа и обновлении весов используется примитив синхронизации `std::shared_mutex`. Потоки инференса захватывают блокировку в разделяемом режиме (shared lock), что позволяет выполнять параллельные предсказания, в то время как процесс загрузки новой модели захватывает эксклюзивную блокировку (unique lock), гарантируя целостность данных в момент переключения сессий.

Листинг 4 (фрагмент) — Выполнение инференса в ONNX Runtime
```cpp
  // Запуск сессии инференса для получения метки класса
  const char* labelOutputName = outputNamePtrs_[0];
  auto outputs = session_->Run(Ort::RunOptions{nullptr}, 
                               inputNamePtrs_.data(), &inputTensor, 1, 
                               &labelOutputName, 1);
```

Полный исходный код модуля инференса, включая механизмы загрузки моделей и управления сессиями, приведён в приложении Б (листинг Б.3).

### 3.4. Управление правилами межсетевого экрана

Подсистема активного противодействия реализована в классе `FirewallManager`, который отвечает за автоматическое блокирование источников вредоносного трафика. В качестве исполнительного механизма используется стандартный межсетевой экран операционной системы Windows (Windows Firewall), управление которым осуществляется через интерфейс командной строки `netsh advfirewall`.

Модуль реализован с использованием паттерна «Одиночка» (Singleton), что гарантирует наличие единственной точки управления правилами фильтрации в системе. Взаимодействие с ОС выполняется асинхронно через класс `QProcess::startDetached`, что исключает блокировку графического интерфейса или потока инференса при ожидании завершения системного вызова. Для предотвращения дублирования правил и снижения нагрузки на стек TCP/IP в классе реализован механизм отслеживания активных блокировок в контейнере `std::set`, что делает операции добавления правил идемпотентными.

Листинг 5 (фрагмент) — Формирование команды блокировки IP
```cpp
  QStringList args;
  args << "advfirewall" << "firewall" << "add" << "rule"
       << "name=DDoS_Block_" + addr.toString()
       << "dir=in" << "action=block"
       << "remoteip=" + addr.toString();

  QProcess::startDetached("netsh", args);
```

В системе также реализован алгоритм «мягкой разблокировки»: при переходе сетевого состояния из «атака» в «норма» система автоматически удаляет все правила с префиксом `DDoS_Block_`, восстанавливая доступ для заблокированных адресов. ### 3.5. Оркестрация: цикл инференса DetectionEngine

Центральным координирующим узлом подсистемы сбора является класс `DetectionEngine`. Он управляет жизненным циклом потока анализа и обеспечивает взаимодействие между захватом пакетов, извлечением признаков и выполнением ML-инференса.

Основной цикл инференса работает в выделенном высокоприоритетном потоке. Работа организована по принципу «оконной агрегации»: в течение 1 секунды система накапливает статистику по всем активным потокам, после чего формируется сводный вектор признаков. Ключевой оптимизацией является использование «порога шума» (Noise Threshold): если суммарная интенсивность трафика на интерфейсе составляет менее 50 пакетов в секунду, система автоматически присваивает окну метку «норма» без вызова нейросетевого классификатора. Это позволяет значительно снизить нагрузку на процессор в периоды сетевого затишья и исключить ложные срабатывания на единичных пакетах.

Листинг 6 (упрощённый фрагмент) — Проверка порога активности и запуск анализа
```cpp
  // Проверка порога активности трафика (NOISE_THRESHOLD_PPS = 50)
  if (result.pps < NOISE_THRESHOLD_PPS) {
    result.label = 0; result.confidence = 0.0f;
  } else {
    auto features = extractor_.computeNormalizedFeatures();
    // Конвертация vector<double> → vector<float> для API ONNX Runtime
    std::vector<float> featFloat(features.begin(), features.end());
    auto [label, conf] = inferencer_.predict(featFloat);
    result.label = label; result.confidence = conf;
  }
```

Управление состоянием защиты реализовано через механизм обратных вызовов (callbacks). При обнаружении атаки оркестратор не только инициирует блокировку через `FirewallManager`, но и отправляет структурированный объект `DetectionResult` в асинхронную очередь базы данных и по сетевому протоколу в графический монитор. Полный исходный код оркестратора `DetectionEngine` приведён в приложении Б (листинг Б.5).

### 3.6. Асинхронная запись в базу данных

Для обеспечения долговременного хранения истории инцидентов и возможности последующего ретроспективного анализа в системе используется РСУБД PostgreSQL. Учитывая высокие требования к пропускной способности (NFR-001), прямое выполнение SQL-запросов `INSERT` в основном потоке обработки пакетов недопустимо из-за высоких задержек дискового ввода-вывода.

В классе `DatabaseManager` реализован паттерн проектирования «Производитель-Потребитель» (Producer-Consumer). Основной поток инференса лишь помещает результаты в неблокирующую lock-free очередь `moodycamel::ConcurrentQueue`, в то время как фоновый поток записи выполняет пакетный сброс данных (batch flush) каждые 5 секунд. Для достижения максимальной скорости записи используется специализированный механизм `COPY` (через интерфейс `stream_to` библиотеки libpqxx), который передаёт данные в бинарном виде, минуя этап парсинга SQL-команд сервером БД. Это позволяет системе сохранять стабильность даже при регистрации тысяч событий в секунду.

Листинг 7 (фрагмент) — Инициализация потока высокопроизводительной записи
```cpp
  pqxx::work txn(conn);
  // Использование механизма COPY для оптимизации записи
  auto stream = pqxx::stream_to::table(txn, {"events"}, 
      {"session_id", "timestamp", "label", "confidence", "pps", 
       "total_packets", "features", "model_name"});
```

Для обеспечения надёжности (требование NFR-005) в модуле реализован механизм «горячего резервирования»: при временной потере сетевого соединения с сервером БД система автоматически переключается на запись событий в локальный файл `FileBuffer`. После восстановления связи данные из файла считываются и дозаписываются в PostgreSQL в хронологическом порядке. Полный исходный код модуля взаимодействия с БД приведён в приложении Б (листинг Б.6).

### 3.7. Обучение модели классификатора

Процесс разработки интеллектуального ядра системы включал этапы сбора данных, конструирования признаков (Feature Engineering), обучения ансамблевых моделей и нейронных сетей, а также их последующий экспорт для высокопроизводительного инференса на языке C++.

**Подготовка набора данных и предобработка**

Для обучения классификаторов использовался открытый набор данных **CIC-DDoS2019**, содержащий записи реального трафика с различными типами современных DoS-воздействий (SYN Flood, UDP Flood, LDAP/DNS Amplification). В ходе подготовки данных в среде Python выполнены следующие операции:
1) **Очистка и нормализация**: удаление дубликатов, обработка пропущенных значений и бесконечных величин (Infinity), возникающих при делении на крайне малую длительность потока.
2) **Балансировка классов**: устранение диспропорции между количеством легитимных пакетов и пакетов атаки методом стратифицированной выборки (stratified sampling) при разбиении на обучающую и тестовую выборки, что обеспечивает репрезентативное представление обоих классов.
3) **Отбор признаков**: корреляционный анализ позволил исключить избыточные переменные, оставив 16 наиболее информативных характеристик, описывающих временну́ю динамику и структуру заголовков.

**Гиперпараметры и выбор моделей**

В рамках исследования были обучены и протестированы три архитектуры:
- **Random Forest (RF)**: конфигурация из 50 решающих деревьев с ограничением глубины до 10 уровней для предотвращения переобучения. Модель показала высокую устойчивость к шумам в данных.
- **Multi-Layer Perceptron (MLP)**: полносвязная сеть с двумя скрытыми слоями (64 и 32 нейрона соответственно) и функцией активации ReLU. Обучение проводилось с использованием оптимизатора Adam и механизма `early_stopping`.
- **XGBoost**: градиентный бустинг с 200 итерациями и скоростью обучения 0,1. Данная модель выбрана в качестве основной благодаря наилучшему балансу между точностью (F1-score = 0,9997) и вычислительной сложностью.

**Экспорт моделей в ONNX**

Модели экспортируются в универсальный формат ONNX с использованием библиотеки `skl2onnx`. При экспорте отключается надстройка `ZipMap` (`zipmap=False`), что позволяет получать вероятности классов в виде прямого тензора `FloatTensor`, читаемого C++ API ONNX Runtime:

Листинг 9 — Экспорт модели в формат ONNX
```python
# Функция экспорта обученной scikit-learn модели в ONNX
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType

def export_onnx_model(model, num_features, filename):
  # Определение типа и размерности входных данных
  initial_type = [('float_input', FloatTensorType([None, num_features]))]

  # Отключение zipmap для получения прямого тензора вероятностей (FloatTensor)
  options = {id(model): {'zipmap': False}}

  # Конвертация модели
  onnx_model = convert_sklearn(model, initial_types=initial_type, options=options)

  # Сохранение сериализованной модели в файл
  with open(filename, "wb") as f:
    f.write(onnx_model.SerializeToString())
```

### 4.6. Ограничения и допущения исследования

В ходе разработки и тестирования программного комплекса были выявлены следующие ограничения, которые следует учитывать при его эксплуатации и дальнейшем развитии:
1) **Протокольная совместимость**: текущая реализация ориентирована исключительно на стек протоколов IPv4. Модули извлечения признаков и управления фаерволом требуют доработки для полноценной поддержки IPv6.
2) **Специфика данных**: обучение моделей проводилось преимущественно на волюметрических атаках (SYN/UDP Flood) из датасета CIC-DDoS2019. Обнаружение сложных атак прикладного уровня (L7), имитирующих поведение человека, может потребовать дополнительного дообучения на специализированных наборах данных.
3) **Аппаратные ограничения**: тестирование производительности проводилось на одиночном вычислительном узле. В условиях магистральных каналов связи с интенсивностью выше 100 000 PPS может потребоваться переход на технологии Zero-copy захвата (DPDK) и горизонтальное масштабирование коллекторов.

В ходе выполнения выпускной квалификационной работы разработан программный комплекс обнаружения атак типа «отказ в обслуживании» (DoS/DDoS), основанный на интеллектуальном анализе статистических характеристик сетевого трафика.

На основании проведённых исследований и разработок сформулированы следующие **основные результаты**:

1) **Проведён комплексный анализ предметной области**: изучены механизмы реализации современных сетевых атак уровней L3 и L4 модели OSI (SYN Flood, UDP Flood, ICMP Flood, Slowloris). Сравнительный анализ существующих решений (Snort, Suricata, Cloudflare, FastNetMon) по 9 критериям показал, что большинство из них либо опираются на статические сигнатуры, либо обладают высокой сложностью настройки. Это обосновало актуальность разработки адаптивной системы на базе ML/DL.

2) **Сформирован и обоснован набор из 16 информативных признаков** сетевых потоков (PPS, byte_rate, syn_packets, entropy и др.), позволяющих эффективно идентифицировать аномальную активность. Обоснован выбор алгоритма градиентного бустинга (XGBoost) и его развёртывания через ONNX Runtime без Python-зависимостей в production.

3) **Спроектирована и реализована распределённая архитектура** системы из двух независимых программ: коллектора `ddos_collector` на C++17 и интерактивной подсистемы мониторинга `ddos_monitor` на Qt6. Разработаны диаграммы 6 ключевых классов, алгоритмы главного цикла (Mermaid-диаграммы) и структура БД из 4 таблиц.

4) **Разработано интеллектуальное ядро** на базе различных архитектур классификаторов (ансамблевых методов XGBoost, RF и нейронной сети MLP), интегрированное через ONNX Runtime с поддержкой горячей замены модели. Задержка инференса составила **0,45 мс** — менее 1 мс на пакет.

5) **Реализована подсистема активного противодействия**: автоматическая блокировка IP-адресов через `netsh advfirewall` с авторазблокировкой. Испытания показали время реакции на инцидент **1,1 с** — в 4,5 раза лучше требования NFR-002 (≤5 с).

6) **Проведено всестороннее тестирование** в соответствии с ГОСТ 19.301-79 [2]. Все 9 функциональных требований выполнены. Точность классификации на эталонном датасете CIC-DDoS2019 составила **F1 = 0,9997 (99,97%)**. Нагрузочные испытания подтвердили стабильную работу при интенсивности до **100 000 пакетов в секунду** с 5-кратным запасом производительности относительно требования NFR-001.

Все задачи, поставленные во введении, решены в полном объёме. Цель работы — разработка эффективного программного средства обнаружения DoS/DDoS-атак — достигнута.

```cpp
void DetectionEngine::processWindow() {
  DetectionResult result;
  result.timestamp = QDateTime::currentDateTime();
  result.modelName = inferencer_.modelName();
  extractor_.fillTelemetry(result); // Сбор телеметрии

  result.droppedPackets = monitor_.droppedPackets();
  result.queueSize = monitor_.queueSize();
  result.blockedIps = FirewallManager::getInstance().getBlockedIps();

  // Пропуск, если пакетов нет
  if (result.totalPackets == 0) {
    if (dumpEnabled_) dumper_.commitWindow(0);
    return;
  }

  // Проверка на порог "шума" (незначительный трафик)
  if (result.pps < NOISE_THRESHOLD_PPS) {
    result.label = 0;
    result.confidence = 0.0f;
    result.features = std::vector<double>(extractor_.featureCount(), 0.0);
    
    if (dumpEnabled_) dumper_.commitWindow(0);
    if (resultCallback_) resultCallback_(result);
    return;
  }

  // Расчет нормализованных признаков
  auto features = extractor_.computeNormalizedFeatures();
  if (features.empty() || (int)features.size() != extractor_.featureCount())
    return;

  result.features = features;
  std::vector<float> featFloat(features.begin(), features.end());

  // Выполнение предсказания ML-моделью
  auto startInfer = std::chrono::steady_clock::now();
  auto [label, confidence] = inferencer_.predict(featFloat);
  auto endInfer = std::chrono::steady_clock::now();

  result.inferenceLatencyMs = std::chrono::duration<double, std::milli>(endInfer - startInfer).count();
  result.label = label;
  result.confidence = confidence;

  // Фиксация окна в дампе трафика
  if (dumpEnabled_) {
    dumper_.commitWindow(result.label);
  }

  // Обновление состояния инцидента и выполнение защитных действий
  updateIncidentState(result);

  // Передача результата подписчикам (GUI, БД)
  if (resultCallback_)
    resultCallback_(result);
}
```

```cpp
void TcpClient::onReadyRead() {
    readBuffer_.append(socket_->readAll());

    while (true) {
        int idx = readBuffer_.indexOf('\n');
        if (idx < 0) break;
        QByteArray line = readBuffer_.left(idx).trimmed();
        readBuffer_.remove(0, idx + 1);
        if (!line.isEmpty())
            processLine(line);
    }
}

void TcpClient::processLine(const QByteArray& line) {
    auto [type, payload] = Protocol::parseMessage(line);

    if (type == Protocol::MSG_STATS) {
        auto result = Protocol::deserializeResult(payload["data"]);
        uint64_t total = payload.value("total_packets", (uint64_t)0);
        emit statsReceived(result, total);
    }
    else if (type == Protocol::MSG_SNAPSHOT) {
        float pps = payload.value("pps", 0.0f);
        uint64_t total = payload.value("total_packets", (uint64_t)0);
        int label = payload.value("current_label", 0);
        emit snapshotReceived(pps, total, label);
    }
    else if (type == Protocol::MSG_BUFFER) {
        std::vector<QByteArray> items;
        if (payload.contains("items")) {
            for (auto& item : payload["items"])
                items.push_back(QByteArray::fromStdString(item.dump()));
        }
        AppLogger::get()->info("Received buffer with {} items.", items.size());
        emit bufferReceived(items);
    }
    else if (type == Protocol::MSG_NOTIFY) {
        QString event = QString::fromStdString(payload.value("event", ""));
        QJsonObject data;
        if (payload.contains("data")) {
            auto doc = QJsonDocument::fromJson(
                QByteArray::fromStdString(payload["data"].dump()));
            data = doc.object();
        }
        emit notificationReceived(event, data);
    }
}
```

```cpp
void DashboardWidget::updateRealtime(const DetectionResult& result, uint64_t totalPackets) {
    addDataPoint(result);
```

```sql
  interface_name TEXT,                -- Имя сетевого интерфейса
  model_name TEXT,                    -- Имя использованной ML-модели
  start_time TIMESTAMP DEFAULT NOW(), -- Время запуска мониторинга
  end_time TIMESTAMP,                 -- Время завершения работы
  total_packets BIGINT DEFAULT 0,     -- Всего обработано пакетов
  total_attacks BIGINT DEFAULT 0,     -- Обнаружено вредоносных окон
  total_benign BIGINT DEFAULT 0       -- Обнаружено легитимных окон
);

-- 2. Таблица событий обнаружения
-- Основное хранилище результатов классификации каждого окна агрегации
CREATE TABLE IF NOT EXISTS events (
  id SERIAL PRIMARY KEY,              -- Уникальный идентификатор события
  session_id INT REFERENCES sessions(id), -- Ссылка на родительскую сессию
  timestamp TIMESTAMP DEFAULT NOW(),  -- Метка времени события
  label INT,                          -- Метка класса (0 - норма, 1 - атака)
  confidence REAL,                    -- Уверенность классификатора (0.0 - 1.0)
  pps REAL,                           -- Интенсивность трафика (пак/сек)
  total_packets BIGINT,               -- Общее число пакетов в окне
  features JSONB,                     -- Вектор признаков в формате JSON
  model_name TEXT                     -- Имя модели, выполнившей инференс
);

-- 3. Таблица снимков статистики
-- Используется для построения графиков в реальном времени
CREATE TABLE IF NOT EXISTS stats_snapshots (
  id SERIAL PRIMARY KEY,              -- Уникальный идентификатор снимка
  session_id INT REFERENCES sessions(id), -- Ссылка на сессию
  timestamp TIMESTAMP DEFAULT NOW(),  -- Метка времени снимка
  packets_per_s REAL,                 -- Текущий PPS
  total_packets BIGINT,               -- Нарастающий итог числа пакетов
  current_label INT                   -- Текущее состояние (атака/норма)
);

-- 4. Таблица инцидентов безопасности
-- Агрегированные данные о выявленных атаках для отчетности
CREATE TABLE IF NOT EXISTS security_events (
  id SERIAL PRIMARY KEY,              -- Уникальный идентификатор инцидента
  session_id INT REFERENCES sessions(id), -- Ссылка на сессию
  start_time TIMESTAMP,               -- Момент начала атаки
  duration_sec REAL,                  -- Продолжительность атаки в секундах
  attacker_ip TEXT,                   -- IP-адрес основного источника атаки
  pps_max REAL,                       -- Пиковая интенсивность за время инцидента
  type_label TEXT,                    -- Тип обнаруженной атаки
  confidence REAL                     -- Средняя/максимальная уверенность
);

-- Индексы для оптимизации выборок по сессиям и времени
CREATE INDEX IF NOT EXISTS idx_events_session_id ON events(session_id);
CREATE INDEX IF NOT EXISTS idx_events_timestamp ON events(timestamp);
CREATE INDEX IF NOT EXISTS idx_security_session ON security_events(session_id);
```

```python
import os
import json
import numpy as np
import pandas as pd
from sklearn.ensemble import RandomForestClassifier
from sklearn.neural_network import MLPClassifier
from xgboost import XGBClassifier
from sklearn.preprocessing import StandardScaler
from sklearn.model_selection import train_test_split
from sklearn.metrics import classification_report, roc_auc_score, confusion_matrix
import onnx
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType
import onnxmltools

# ==============================================================================
# Файл: train_model.py
# Описание: Скрипт обучения и экспорта моделей обнаружения DoS-атак.
# Реализует пайплайн предобработки данных CIC-DDoS2019 и конвертацию в ONNX.
# ==============================================================================

# Признаки трафика (согласно FeatureExtractor.cpp)
FEATURES = [
  "flow_duration", "total_fwd_packets", "total_bwd_packets",
  "total_len_fwd", "total_len_bwd", "fwd_mean", "bwd_mean",
  "flow_pps", "syn_packets", "ack_packets", "fin_packets",
  "rst_packets", "psh_packets", "urg_packets", "entropy", "avg_window_size"
]

def load_data(path):
  """Загрузка CSV-файлов датасета CIC-DDoS2019"""
  df = pd.read_csv(path, low_memory=False)
  df.columns = df.columns.str.strip()
  # Очистка бесконечных и пропущенных значений
  df.replace([np.inf, -np.inf], np.nan, inplace=True)
  df.dropna(inplace=True)
  # Бинаризация меток: BENIGN -> 0, Остальное -> 1
  df['Label'] = df['Label'].apply(lambda x: 0 if 'BENIGN' in str(x).upper() else 1)
  return df

def save_scaler(scaler, filename, use_log1p=False):
  """Сохранение параметров нормализации в формате JSON для C++ ядра"""
  params = {
  "features": FEATURES,
  "mean": scaler.mean_.tolist(),
  "scale": scaler.scale_.tolist(),
  "use_log1p": use_log1p
  }
  with open(filename, 'w', encoding='utf-8') as f:
  json.dump(params, f, indent=4)

def export_onnx(model, num_features, filename, is_xgb=False):
  """Конвертация обученной модели в универсальный формат ONNX"""
  initial_type = [('float_input', FloatTensorType([None, num_features]))]
  if is_xgb:
  onnx_model = onnxmltools.convert_xgboost(model, initial_types=initial_type)
  else:
  # Отключение ZipMap для получения прямого тензора вероятностей
  options = {id(model): {'zipmap': False}}
  onnx_model = convert_sklearn(model, initial_types=initial_type, options=options)

  with open(filename, "wb") as f:
  f.write(onnx_model.SerializeToString())

def main():
  # Загрузка и разделение выборки (80/20)
  df = load_data("cicddos2019_sample.csv")
  X = df[FEATURES].values
  y = df['Label'].values
  X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2, stratify=y)

  # Обучение XGBoost (основная модель)
  scaler = StandardScaler()
  X_train_s = scaler.fit_transform(X_train)
  model = XGBClassifier(n_estimators=200, learning_rate=0.1, max_depth=6)
  model.fit(X_train_s, y_train)

  # Сохранение артефактов
  save_scaler(scaler, "xgb_scaler_params.json")
  export_onnx(model, len(FEATURES), "xgb_model.onnx", is_xgb=True)

  # Вывод итоговых метрик
  y_pred = model.predict(scaler.transform(X_test))
  print(classification_report(y_test, y_pred))

if __name__ == "__main__":
  main()
```
