# **ВВЕДЕНИЕ**

С давних времён человечество использовало печати и подписи для подтверждения подлинности документов и сообщений. Точная принадлежность сообщения позволяет быть уверенным в его подлинности, а также в том, что человек, подписавший сообщение, несёт за его содержание ответственность. Таким образом, устанавливаются доверительные отношения между отправителем и получателем.

С приходом цифровой эры и появлением новых видов носителей информации возник вопрос об их сохранности при доставке и верификации при получении другим пользователем. Для решения этих проблем есть два пути: шифрование сообщения и дополнение его электронной цифровой подписью (ЭЦП). Оба варианта предусматривают то, что получатель имеет необходимые средства для расшифровки сообщения/его подписи. Однако, оба способа служат разным целям. Шифрование используется для сокрытия сообщения от посторонних глаз, тогда как ЭЦП направленна именно на следующие цели:

* Неотказуемость (автор несёт за содержание сообщения полную ответственность);
* Целостность (невозможность изменить подписанный документ);
* Авторство (устанавливается чёткое соответствие между автором и сообщением);

Из подобной концепции получается главное преимущество ЭЦП над шифрованием: при работе с большими файлами ЭЦП требует гораздо меньших ресурсных затрат, поскольку допускает шифрование не всего файла, а лишь его одностороннее преобразование, представляющее из себя малый объём данных и не поддающееся восстановлению.

Второе преимущество происходит из схемы построения ЭЦП. Как правило она представляет из себя ограниченное стандартом число байт, добавляемых в файл, тогда как криптосистемы шифрования такой же сложности вызывают сильное увеличение объёма конечного сообщения, иногда на несколько порядков.

# **ТЕОРЕТИЧЕСКИЕ ОСНОВЫ**

## **Описание предметной области**

Электронная цифровая подпись – это реквизит электронного документа или сообщения, получаемый криптографическим преобразованием содержания документа и сертификата пользователя, проверяющий этот документ на искажение содержащейся в нём информации с момента подписания и принадлежность этого документа автору.

Проверка на искажение достигается любым односторонним преобразованием сообщения, после чего результат преобразования отсылается по закрытому каналу отдельно от сообщения. Однако, проверка на искажение не обеспечивает безопасности полученной информации. Пусть информация и является целостной, её авторство не всегда возможно установить, чем может воспользоваться третье лицо. Сообщение должно обладать свойством аутентичности, для чего в алгоритмах ЭЦП для каждого пользователя назначаются ключи подписи, только при наличии которых можно проводить процедуру проверки файла на отсутствие искажений.

Как уже было сказано выше, основные цели, которым служит ЭЦП, являются неотказуемость, авторство и целостность. Для достижения первых двух алгоритмы цифровых подписей должны опираться на следующие принципы:

* Алгоритм должен быть общеизвестен;
* Только автор должен обладать возможностью поставить свою подпись;
* Потенциальный получатель должен обладать способом проверить подпись, но не должен обладать возможностью поставить подпись от лица отправителя;
* Если подпись может быть подтверждена получателем, то не должно быть сомнений в авторстве отправителя;

Первые алгоритмы ЭЦП берут своё начало из алгоритмов шифрования для передачи сообщений по незащищённым каналам связи, таких как протокол Диффи-Хеллмана. Алгоритм позволял переслать секретный ключ по незащищённым каналам связи. В 1976 году Уитфилд Диффи и Мартин Хеллман предположили существование ЭЦП, алгоритм которого уже годом позже был разработан на основе схемы шифрования RSA и используется до сих пор.

Схема асинхронного шифрования RSA, как и протокол Диффи-Хеллмана, предполагала, что для пересылки сообщения по открытому каналу, расшифровка сообщения должна быть гораздо сложнее, чем его расшифровка, поэтому для шифрования сообщения используется открытый ключ, который можно пересылать по открытому каналу связи, а для расшифровки – закрытый, который необходимо хранить в тайне. Для схемы цифровой подписи используется тот же принцип, однако задача шифрования должна быть более лёгкой задачей, чем расшифровка, поэтому закрытый ключ должен оставаться у отправителя, а открытый становится общедоступен и используется для расшифровки. Такая схема работы ЭЦП используется практически во всех современных ЭЦП.

## **Виды алгоритмов ЭЦП**

Алгоритмы цифровой подписи можно классифицировать по типу ключей, используемой односторонней функции и по алгоритму шифрования.

По типу ключей алгоритмы можно разделить на два вида – на симметричном и ассиметричном шифровании (шифровании с открытым ключом). В симметричных схемах для шифрования и расшифровки используется один и тот же ключ, который должен быть у обеих сторон, но должен быть секретным. В асимметричных схемах секретный ключ должен быть у автора, а открытый доступен всем. Рассмотрим достоинства и недостатки обеих систем.

Плюсы ЭЦП на симметричных шифрах:

* Малые вычисления;
* Основаны на блочных шифрах, высокая стойкость которых доказана;
* Простая технологическая масштабируемость;

Минусы:

* Большая длина подписи из-за того, что необходимо подписывать каждый байт (из-за особенностей блочных шифров);
* Необходимость менять ключи для установки и проверки каждой новой подписи;
* Необходимость хранить много ключей сразу;
* Требуется использование третьего лица для раздачи ключей;

Плюсы ЭЦП на асимметричных шифрах:

* Малая длина ЭЦП;
* Существование отдельной пары открытого и закрытого ключей (закрытый ключ не нужно передавать, а по открытому ключу невозможно вычислить закрытый за разумное время);

Минусы:

* Сложные вычисления;

Существование алгоритмов ЭЦП на симметричном шифровании было предположено создателями алгоритма RSA ещё в семидесятых, однако из-за данных недостатков, до сих пор не разработано ни одного достаточно эффективного алгоритма цифровых подписей на основе симметричных шифров.

По типу односторонней функции алгоритмы делятся на алгоритмы, использующие шифры с секретом, использующие хеш-функции и не использующие никакие. Современные ЭЦП используют хеш-функции, на основе выходных значений которых создаётся и проверяется подпись. Выходное значение хеш-функции всегда одной длины, что удобно при разработке алгоритма. Более ранние алгоритмы использовали шифры с секретом, но они были уязвимы, так как секрет можно было узнать по открытому ключу, который доступен всем. К тому же, шифры с секретом не гарантируют одинаковую длину выходного значения. Не использование никаких функций ведет к низкой надёжности криптосистем, а те криптосистемы, что возможны без использования односторонних функций, вычисляют подписи очень большой длины.

Видов алгоритмов по используемому алгоритму шифрования очень много, однако современные шифры, по большей части, базируются на сложности расчёта уравнений в группе точек эллиптической кривой над замкнутым полем.

## **Постановка задачи**

Задача этой выпускной квалификационной работы состоит в следующем: необходимо написать комплекс из двух независимых программ для реализации алгоритмов ЭЦП на основе стандарта ГОСТ 38.10-2018. Программы предназначаются для работников предприятия, желающего обезопасить своих работников от неподписанного ПО.

Первая программа реализует алгоритм генерации электронной подписи на основе файла и данных параметров, должна генерировать открытый и закрытый ключи шифрования. Вторая программа реализует алгоритм проверки электронной подписи файла на основе данных параметров и открытого ключа шифрования.

Первая программа представляет из себя приложение, реализующее несколько функций:

* Вывод справки;
* Генерация новой пары ключей и их сохранение;
* Генерация ЭЦП на основе пары ключей, параметров и содержания файла, добавление ЭЦП к файлу;
* Вывод сообщения об успешном завершении работы или информацию о возникшей ошибке;

Алгоритм работы пользователя получается следующим: если пользователь ещё не имеет пары ключей, то он должен их сгенерировать, указав своё имя. Далее он должен указать своё имя и путь до файла (относительный или абсолютный), который необходимо подписать.

Вторая программа реализует функцию проверки подписи и вывод сообщения о результатах проверки с именем владельца файл, а также информации о возникших ошибках. Она автоматически открывает каждый исполняемый файл, открываемый пользователем.

Программы комплекса должны быть разработаны на языке высокого уровня C с использованием библиотеки GNU Multi‑Precision Library на ОС Ubuntu или любую другую ОС на базе ядра Linux. Для запуска второй программы должен использоваться скрипт для командной оболочки среды Bash.

## **Аналогичные разработки**

Среди аналогичных разработок следует отметить средство Microsoft Windows для работы с электронной цифровой подписью исполняемых файлов. Данное средство предназначено для показа диалогового окна пользователю, открывающему исполняемые файлы. Диалоговое окно содержит информацию о владельце файла и предоставляет пользователю выбор: запускать или не запускать этот файл. Данное средство способно работать со своими форматами сертификатов ЭЦП, в которых помимо параметров и ключей содержатся также наименование стандарта ЭЦП и стандарта хеш-функции.

Для ОС на базе ядра Linux и для ОС Windows существует множество решений для формирования и проверки целостности и авторства сообщений, файлов и пакетов из репозиториев, но среди них нет решений, позволяющих проводить такую проверку автоматически, при запуске любого приложения в системе. Исключением является лишь вышеупомянутое средство для ОС Windows.

## **Описание используемых алгоритмов**

Используемый в данной работе алгоритм цифровой подписи соответствует стандарту ГОСТ 34.10-2018, который основывается на сложности операций в группе точек эллиптической кривой, определённой над конечным полем. Для обеспечения неизменяемости сообщения используется хеш-функция стандарта ГОСТ 34.10-2018 «Стрибог». Оба стандарта являются международными и основаны соответственно на российских национальных стандартах ГОСТ Р 34.10-2012 и ГОСТ Р 34.11-2012.

Электронную подпись можно хранить как отдельно, так и в составе сообщения или файла. Данный стандарт описывает способ хранения подписи в конце сообщения.

![](data:image/png;base64...)

Рис. 1. Структура файла и подписи в нём

Подобный способ расположения подписи в файле порождает некоторые последствия. Некоторые форматы данных (например, .doc файлы) становятся недоступны для открытия. При чтении текстовых файлов с подписью, текст подписи будет отображаться в конце файла как случайный набор символов, что может затруднить восприятие информации. Данных проблем можно избежать, если использовать заранее подготовленное ПО, удаляющее последние несколько байт информации, соответствующих длине подписи, или читающее весь файл, кроме подписи. В случае исполняемых файлов проблем их исполнения возникать не должно, поскольку информация о подписи располагается после инструкции завершения программы и не затрагивает оригинальную информацию.

Стандарт ЭЦП использует шифрование на эллиптической кривой над конечным простым полем Fp [3]. Формула кривой: y2≡x3+ax+b(mod p), где p – простое число, x, y, a, b ∈ Fp, а 4a3+27b2(mod p)≠0 и 1728. Процессы сложения и умножения точек на кривой разобраны в пункте 2.5, часть которого посвящена разработке алгоритмов для реализации данных процессов.

Электронная цифровая подпись имеет набор параметров, используемый в алгоритмах формирования и проверки подписи. Для успешной проверки подписи все параметры должны быть такими же, что и при формировании (при проверке недоступен лишь закрытый ключ d).

Первым параметром является вышеупомянутый p – модуль эллиптической кривой. Рекомендуемый размер p>2255 для длины подписи 32 байта и p>2512 для длины подписи 64 байта. Параметры a и b ∈ Fp – коэффициенты кривой такие, что инвариант кривой, задаваемый ими, не 0 и не 1728. Так же параметрами являются число m – порядок всех точек кривой (т.е. их количество), число q – порядок некоторой циклической подгруппы, точка P≠O такое, что q\*P=O, где O – нулевой элемент кривой. Подробнее особенности реализации арифметических операций разобраны в пункте 2.5. Порядок подгруппы всегда выполняет равенство q=m\*n, где n – целое число. Для длины подписи 32 байта 2254<q<2256, а для 64 байт 2508<q<2512.

Последними параметрами ЭЦП, а также ключами к ней являются параметры d и Q. Целое число d – это случайное число от 1 до q-1 и закрытый ключ, использующийся для формирования цифровой подписи. Поэтому от размеров p и q зависит не только криптостойкость системы, но и количество возможных пользователей. Точка Q≠O является результатом произведения d и P. Точка Q является открытым ключом и должна быть известна всем тем, кто хочет проверить подпись и установить её владельца.

В данной работе не рассматривается вопрос формирования параметров цифровой подписи и эффективного способа их хранения. Вместо этого используются параметры, описанные в примерах оригинального стандарта ЭЦП. Рассматривается лишь процесс формирования ключей.

### **Алгоритм формирования подписи**

В данной работе производится работа с подписями длиной 64 байта или 512 бит. ЭЦП состоит из векторов r и s, расположенных последовательно друг за другом в таком порядке от старшие разрядов к младшим.

Важно отметить преобразование векторов в целые числа и обратно. Например, для вектора хеш-функции h(hn-1, …,h0) существует соответствующее ему число ![](data:image/x-wmf;base64...).

Формирование подписи происходит в несколько этапов:

Этап 1 – нахождение числа e ≡ α (mod q), где α – число, соответствующее вектору хеш-функции сообщения. Если e = 0, то e принимается равной 1.

Этап 2 – генерация случайного числа k, 0 < k < q.

Этап 3 – вычисление координат точки C, где C = kP.

Этап 4 – получение чисел r и s, r ≡ xc (mod q), s ≡ rd + ke (mod q). Если r или s равно 0, то необходимо вернуться ко 2 шагу.

Этап 5 – получение векторов, соответствующих векторам r и s, конкатенация этих векторов.

### **Алгоритм проверки подписи**

Проверка хеш-функции осуществляется в такой последовательности:

Этап 1 – получение хеш-функции сообщения. Важно учитывать, что хеш-функция вычисляется из содержания сообщения без подписи.

Этап 2 – получение подписи файла (последних 64 байт), разделение её на 2 вектора и получение числе r и s, соответствующих этим векторам.

Этап 3 – получение числа e ≡ α (mod q), где α – число, соответствующее вектору хеш-функции сообщения. Если e = 0, то e принимается равной 1.

Этап 4 – получение ν ≡ e-1 (mod q).

Этап 5 – вычисление z1 ≡ sν (mod q) и z2 ≡ -rν (mod q).

Этап 6 – вычисление координат точки C, где C=z1P+z2Q.

Этап 7 – получение числа R ≡ xc (mod q). Только если R=r, то подпись действительная, иначе она неверна.

Под неверной подписью понимается 3 состояния: неподходящие параметры, отсутствие нужных открытых ключей и отсутствие подписи.

Для получения массива строк команда в консоли разбивается на строки символом пробела, поэтому нулевым элементом массива выступает строка вызова программы «sds\_gen». С помощью функций GMP для целых чисел переменной длины mpz\_init() инициализируются переменные, отвечающие за параметры ЭЦП и ключи. Затем параметры ЭЦП загружаются из файла ds\_params в режиме чтения с помощью функции GetParams, в которой последовательно загружаются значения p, a, b, m, q, xP и yP, полученные значения выводятся. Если при открытии файла возникла ошибка, то выводится сообщение и функция возвращает «-1». Выполняется проверка полученных значений. Если хотя бы один из параметров меньше или равен 0, то значит, что файл параметров был сгенерирован неправильно, был повреждён или не соответствует указанной в пункте 2.2 структуре. Выводится сообщение об ошибке, и функция возвращает «-1». В нормальной ситуации функция возвращает «0». Возвращаемое значение функции GetParams сохраняется в переменной parfstatus.

Инициализация переменных с помощью GMP работает на основе функции выделения памяти malloc(), которая не подразумевает автоматической очистки памяти даже после закрытия приложения, поэтому разработана функция Clear\_GMP, последовательно очищающая параметры ЭЦП, выделенные при их получении. Очистка памяти, выделенной под целые числа, производится вызовом функции mpz\_clear(). Данную операцию необходимо проделывать перед окончанием работы программы, т.е. перед окончанием работы каждой команды, кроме команды по умолчанию и вывода справки, потому что они выполняются до любых операций выделения памяти.

Вывод в файл значений ключей (которые являются целыми числами произвольной длины) в файл осуществляется функциями GMP mpz\_out\_str(указатель на файл, разрядность, переменная). Алгоритм проверки целостности файла заключается в проверке количества строк. Для этого файл построчно считывается в буфер (buffer) до получения EOF, на каждом считывании к счётчику (num) прибавляется единица. Условием прохождения проверки на целостность является равенство num%4=1 для файла accounts и n%3=1 для файла private\_accounts. После прохождения проверки необходимо открыть файл заново, чтобы повторить цикл чтения/записи.

Перед поиском пользователя необходимо инициализировать переменную loginfound = 0. Поиск осуществляется считыванием каждой строки файла в буфер, пока не достигнут EOF или loginfound не станет равной 1. Тогда на итерациях, в которых в буфер записывается логин (login) (условие num%4=0 для accounts и num%3=0 для public\_accounts), каждый символ буфера проверяется на совпадение с символом переноса строки (условие buffer[i]==0x0A). Если условие выполняется, то символ в буфере заменяется на 0. После проверки всех символов в буфере, он сравнивается с полученным из аргументов логином (login). Если строки равны, то переменная loginfound = 1. Условием нахождения пользователя является значение loginfound == 1 и num != 0.

Функция GetUserKeys работает схожим образом. Открывается файл accounts для чтения. В случае ошибки выводится соответствующее сообщение. Осуществляется поиск по введённому логину в файле accounts. Если пользователь не найден, то выводится сообщение об ошибке, иначе считываются следующие 3 строки после найденного логина соответственно в переменные d, xQ и yQ. Проверяется, что все три переменные больше нуля, иначе выводится сообщение об ошибке.

Работа функции GenerateDS заключается в реализации алгоритма формирования цифровой подписи. Цифровая подпись длиной 64 байта состоит из двух векторов, соответствующих числам r и s (r, s ≠ 0), длиной по 32 байта, которые соединяются последовательно. Находится случайное число k, которое умножается на точку P. Координата x результирующей точки по модулю q является вектором r. Вектор s находится как rd+ke, где e =α(mod p). Число α это число, соответствующее вектору хеш-функции.

Сборка первой программы осуществляется в 3 этапа: компиляция объектов модулей hash\_lib.c и ellipt\_cur.c, компиляция объекта основной программы, сборка (связывание) исполняемого файла программы из полученных объектов (машинного кода).

Первая программа требует существование работающих конфигурационных файлов в директории /usr/local/etc/sds/. Поэтому Makefile будет выглядеть так:

.PHONY: any clean install uninstall

any: output

output: sds\_gen.o elipt\_cur.o hash\_lib.o

    gcc sds\_gen.o elipt\_cur.o hash\_lib.o -o sds\_gen -lgmp

elipt\_cur.o: elipt\_cur.c

hash\_lib.o: hash\_lib.c

    gcc -c hash\_lib.c -lgmp

clean:

    rm \*.o sds\_gen

install:

    install ./sds\_gen /usr/local/bin

    install -d /usr/local/etc/sds

    install ./ds\_params /usr/local/etc/sds

    install ./accounts /usr/local/etc/sds

    install ./public\_accounts /usr/local/etc/sds

uninstall:

    rm -rf /usr/local/bin/sds\_gen

Целью по умолчанию является цель «any», которая не требует указания при введении команды «make». Эта цель зависит от выполнения цели «output», которая зависит от целей «sds\_gen.o», «ellipt\_cur.o», «hash\_lib.o» (по сути их существование в той же директории, что и Makefile). Цель «output» выполняет сборку исполняемого файла sds\_gen.

Цель «sds\_gen.o» зависит от файлов «sds\_gen.c», «ellipt\_cur.h» и «hash\_lib.h». Выполняется инструкция компиляции объекта sds\_gen.o. Заголовочные файлы нужны при препроцессинге, поскольку содержат внешние функции, используемые программой. Цели «ellipt\_cur.o» и «hash\_lib.o» получают машинный код из файлов elipt\_cur.c и hash\_lib.c. При компиляции и сборке используется флаг -lgmp для включения в сборку функции из библиотеки GMP.

Цель «clean», выполнение которой выполняется введением команды «make clean», удаляет из директории все объекты, а также исполняемый файл sds\_gen.

Цель «install», выполнение которой также требует аргумента install для команды make копирует исполняемый файл sds\_gen в директорию /usr/local/bin, создаёт директорию /usr/local/etc/sds, и копирует туда файлы ds\_params, accounts и public\_accounts. Цель «uninstall» удаляет скопированный sds\_gen и директорию sds.

Таким образом, для сборки программы требуется ввести команду «make», для установки требуется ввести команду «make install», а для удаления требуется ввести «make uninstall». Конечному пользователю можно передавать как исходный код, так и готовую для установки сборку, поскольку цели «install» и «uninstall» не зависят от наличия в директории файлов с исходным кодом.

## **Разработка второй программы**

Вторая программа, по сути своей, не является одним исполняемым файлом. Вместо этого, она реализуется в виде динамической библиотеки libsds\_ver.so, в которой переопределяется системный вызов линукса, отвечающий за открытие исполняемых файлов execve, принимающий как аргументы строки с аргументами для команды, путём до этого исполняемого файла и параметры окружения. Смысл такого действия заключается в выполнении проверки программы на наличие подписи перед её настоящим выполнением. Программы в Linux можно запускать из терминала, для которого и происходит переопределение системного вызова.

Переменную окружения LD\_PRELOAD можно определить и как локальную, и как глобальную, что будет значить, что она будет релевантной до тех пор, пока не будет удалена из системы вручную. После определения переменной для целевой программы эта программа будет запущена. Создаётся Bash-скрипт sds\_ver\_auto с флагом исполняемости, определяющий переменную:

LD\_PRELOAD = libsds\_ver.so gnome-terminal

При выполнении скрипта запустится экземпляр терминала gnome-terminal по умолчанию для ОС Ubuntu последних версий для которого вызов любого другого исполняемого файла заменится функцией из библиотеки libsds\_ver.so. Это безопасно, поскольку при LD\_PRELOAD в данном случае является локальной переменной для данного экземпляра приложения [1], не действует больше нигде в системе и пропадает вместе с закрытием терминала. Можно изменить терминал, изменив его название в Bash-скрипте. Потенциально возможно предзагрузить библиотеку и перед запуском оболочки системы, но это может привести к краху всей системы в случае непредвиденной ошибки. Приведённый выше код можно вводить и вручную, однако проще запускать готовый скрипт.

Флаг исполняемости для скрипта необходим по той причине, что Bash-скрипты без флага запускаются с помощью команды «bash <имя файла>» и требуют наличия определённых прав доступа для исключения ошибки прав доступа в ходе их выполнения. Для выполнения задач, определяемых во второй программе, таких прав не требуется (исключение составляют лишь те случаи, когда администратор запретил запуск терминала для определённых групп пользователей).

Предзагруженная библиотека повторяет интерфейс оригинального системного вызова [6], однако обладает своим алгоритмом действий. В начале открывается целевой исполнительный файл (его путь является одним из аргументов, передаваемых в execve). Если его открыть не удалось, то выводится ошибка. Иначе, получается id пользователя системы. Если id=0, значит пользователь, создавший файл, находился под root и является администратором системы. Если id≠0, то выполняется функция Verifier, проверяющая подпись файла. Если пользователь не найден и результат -1, то выводится сообщение с предупреждением о том, что не удалось установить владельца, на которое нужно ответить Y/N (да или нет) в любом регистре. Если ответ N (нет), то основная функция возвращает NULL. Если пользователь ответил «да», id=0 или файл открыть не удалось, то функция возвращает вызов оригинальной функции execve, которая находится по имени следующей после переопределённой в файле ld.so и проводится выполнение оригинального файла.

Функция Verifier входит в состав модуля sds\_ver.c, подключаемого к основной библиотеке. Эта функция инициализирует переменные параметров, как целые числа, а затем вызывает функцию GetParams, которая работает точно так же, как и в первой программе. Аналогично, там присутствует функция ClearGMP, очищающая переменные GMP, являющиеся параметрами и ключами ЭЦП. Результат выполнения GetParams записывается в переменную parfstatus. Если parfstatus = 0, то продолжается работа, иначе выводится ошибка отсутствия параметров и возвращается значение «-1». Если parfstatus = 0, то открывается целевой файл. В случае ошибки выводится сообщение «файл открыть не удалось», очищаются переменные и возвращается значение «-1». Производится проверка на пустоту файла. Указатель в файле переносится в конец и с помощью функции ftell() полчается разница между началом файла и текущим положением указателя, что и является размером файла. Указатель снова перемещается в начало файла. Если файл пуст, то вызывается ошибка, очищается память и возвращается «-1». Иначе вызывается функция CheckDS, реализующая алгоритм проверки ЭЦП, результат записывается в переменную result. Память освобождается, значение result возвращается. Это необходимо по той причине, что при генерации хеш-функции во время формирования подписи, подпись не входила в состав файла, она была добавлена позже.

Функция GetRSFromFile предназначена для получения векторов r и s длины по 32 байта, входящих в состав подписи и их преобразования в целые числа. Для этого указатель в файле перемещается в конец со смещением 64 байта влево, считываются 32 байта из файла и с помощью функции библиотеки GMP mpz\_import получается целое число r. Затем аналогично получают число s.

Как и функция получения хеш-функции, функция GetUserKeys имеет иной алгоритм. Открывается файл public\_accounts. В случае ошибки выводится соответствующее сообщение и возвращается значение «-1». Иначе проверяется условие, что указатель не указывает на симвок конца файла (EOF) с помощью функции feof(). Если указывает, то выводится соотвествующая ошибка и возвращается «-1». Считывается 3 строки в буфер в цикле от 0 до номера текущей итерации, который передаётся через аргументы. Затем считывается ещё одна строка. В цикле от 0 до длины буфера проверяется каждый символ на равенство символу переноса строки 0x0A. Если символ равен символу переноса строки, то он приравнивается 0. После проверки, символ записывается в строку login. С помощью функции mpz\_inp\_str загружаются значения xQ и yQ. Если xQ или yQ меньше или равны 0, то выводится сообщение об повреждении файла public\_accounts, возвращается значение «-1». В случае успешной работы возвращается значение «0».

Цели «ellipt\_cur.o» и «hash\_lib.o» получают машинный код из файлов elipt\_cur.c и hash\_lib.c. При компиляции и сборке используется флаг -lgmp для включения в сборку функции из библиотеки GMP. Поскольку в результате необходимо получить динамическую библиотеку, то все объекты необходимо компилировать как позиционно-независимый код, для чего используется флаг -fPIC.

Цель «clean», выполнение которой выполняется введением команды «make clean», удаляет из директории все объекты, а также библиотеку libsds.so.

Цель «install», выполнение которой также требует аргумента install для команды make копирует исполняемый файл sds\_gen\_auto в директорию /usr/local/bin, создаёт директорию /usr/local/etc/sds, и копирует туда файлы ds\_params и public\_accounts, копирует бибилиотеку libsds.so в директорию /usr/local/lib/. Цель «uninstall» удаляет sds\_ver\_auto, библиотеку libsds.so и директорию sds.

Таким образом, для сборки программы требуется ввести команду «make», для установки требуется ввести команду «make install», а для удаления требуется ввести «make uninstall».

Реализованный комплекс решает задачи формирования и проверки цифровой подписи, формирования ключей на основе готовых параметров, реализована автоматическая проверка каждого запускаемого исполняемого файла, не принадлежащего администратору системы (а заодно не принадлежащего самой системе).

**СОДЕРЖАНИЕ РАЗДЕЛОВ ЗАДАНИЯ И ИСХОДНЫЕ ДАННЫЕ**

Необходимо разработать программный комплекс, предназначенный для подписания файла, проверки подлинности подписи и установления личности, подписавшей файл. Таким образом, должна обеспечиваться неизменность подписанного файла, а также установление ответственного лица, подписавшего файл. Алгоритм электронной цифровой подписи должен соответствовать стандарту ГОСТ 34.10-2018. Комплекс состоит из двух отдельных независимых программ.

Первая программа комплекса должна реализовывать генерацию подписи с помощью приватного ключа, принадлежащего только лицу, которому необходимо подписать файл. Программа также должна добавлять подпись в файл.

Вторая программа должна проверять файлы, используемые конечным пользователем, на предмет наличия у них подписи. В случае, если подпись верна, должно устанавливаться лицо, подписавшее файл и разрешаться дальнейшее использование файла. В случае, если подпись не верна или отсутствует, необходимо установить, действительно ли пользователь хочет работать с данным файлом во избежание проблем с безопасностью. Правильность подписи, а также личность подписавшего может быть установлена только в случае, если у конечного пользователя есть публичный ключ для расшифровки подписи, который передаётся пользователю отдельно.

Разработка должна обеспечиваться средствами языка высокого уровня C для работы в операционных системах на базе ядра Linux.

int SaveKeys (char\* login, mpz\_t d, mpz\_t xQ, mpz\_t yQ)

{

    FILE \*accounts, \*public;

    char loginfound = 0;

    char buffer[256];

    int num = 0;

    if ((accounts = fopen("/usr/local/etc/sds/accounts", "rb+")) == NULL)

    {

        printf("Error. File \"accounts\" not found\n");

        return -1;

    }

    if ((public = fopen("/usr/local/etc/sds/public\_accounts", "rb+")) == NULL)

    {

        printf("Error. File \"public\_accounts\" not found\n");

        return -1;

    }

    while (feof(accounts) == 0 )

    {

        fgets(buffer, 256, accounts);

        num ++;

    }

    if (num % 4 != 1)

    {

        printf("Accounts file is corrupted\n");

        fclose(accounts);

        fclose(public);

        return -1;

    }

    {

        printf("Error reading accounts info\n");

        return -1;

    }

    num = 0;

    while (feof(public) == 0 )

    {

        fgets(buffer, 256, public);

        num ++;

    }

    {

        printf("Public accounts file is corrupted\n");

        fclose(accounts);

        fclose(public);

        return -1;

    }

    num = 0;

    while (feof(accounts) == 0 && loginfound == 0)

        }

        num++;

    }

    {

        printf("Error. User %s already exists\n", login);

        fclose(accounts);

        fclose(public);

        return -1;

    }

    if ((public = freopen("/usr/local/etc/sds/public\_accounts", "rb+", public)) == NULL)

    {

        printf("Error reading public accounts info\n");

        return -1;

    }

    num = 0;

    while (feof(public) == 0 && loginfound == 0)

    {

        fgets(buffer, 256, public);

        if (num % 3 == 0)

        {

            for (int i=0; i<256; i++)

            {

                if (buffer[i] == 0x0A)

                    buffer[i]=0;

            }

            if (strcmp(buffer, login) == 0)

                loginfound = 1;

        }

        num++;

    }

    if (loginfound == 1 && num != 0)

    {

        printf("Error. User %s already exists\n", login);

        fclose(accounts);

        fclose(public);

        return -1;

    }

    fseek(accounts, 0, SEEK\_END);

    fputs(login, accounts);

    fputs("\n", accounts);

    mpz\_out\_str(accounts, 16, d);

    fputs("\n", public);

    mpz\_out\_str(public, 16, yQ);

    fputs("\n", public);

    printf("Keys generated!\n");

//генератор ЦП

int GenerateDS (mpz\_t p, mpz\_t a, mpz\_t b, mpz\_t m, mpz\_t q, mpz\_t xP, mpz\_t yP, mpz\_t d, mpz\_t xQ, mpz\_t yQ, unsigned char \*ds, FILE \*file)

{

    mpz\_t e, alpha, k, xC, yC, r, s, tmp;

    gmp\_randstate\_t state;

    unsigned char s\_str[32], r\_str[32];

    unsigned char \*h;

    mpz\_init(e);

    mpz\_init(alpha);

    mpz\_init(k);

    mpz\_init(xC);

    mpz\_init(yC);

    mpz\_init(r);

    mpz\_init(s);

    mpz\_init(tmp);

    gmp\_randinit\_default(state);

    gmp\_randseed\_ui(state, time(NULL));

    //получение хеш-кода

    h = GenerateHashFromFile(file);

    //получить альфа, число, двоичным представлением которого является h

    mpz\_import(alpha, 32, 1, 1, 1, 0, h);

    gmp\_printf("alpha = %Zx\n", alpha);

    //получить e

    mpz\_mod(e, alpha, q);

    if (mpz\_cmp\_si(e, 0) == 0)

        mpz\_set\_si(e, 1);

    //генерировать k, если r или s равны 0

    while (mpz\_cmp\_si(r, 0) == 0 || mpz\_cmp\_si(s, 0) == 0)

    {

        //получить k

        mpz\_urandomb(k, state, 256);

        gmp\_printf("k = %Zx\n", k);

        //C=k\*P

        PointMul(p, a, xP, yP, k, xC, yC);

        //r=xCmodq

        mpz\_mod(r, xC, q);

        //для того, чтобы пропустить ненужные шаги, если r=0

        if (mpz\_cmp\_si(r, 0) != 0)

        {

            //s=(rd+ke)modq

            mpz\_mul(tmp, r, d);

            mpz\_mul(s, k, e);

            mpz\_add(s, s, tmp);

            mpz\_mod(s, s, q);

            //для того, чтобы пропустить ненужные шаги, если s=0

            if (mpz\_cmp\_si(s, 0) != 0)

            {

                //преобразование r и s в строки и получение строки цифровой подписи=(r||s)

                mpz\_export(r\_str, NULL, 1, 1, 1, 0, r);

                mpz\_export(s\_str, NULL, 1, 1, 1, 0, s);

                for (int i=0; i<32; i++)

                {

                    ds[i]=r\_str[i];

                    ds[i+32]=s\_str[i];

                }

                printf("ds = ");

                for (int i = 0; i<64; i++)

                    printf("%x ", ds[i]);

                printf("\n");

            }

        }

    }

    free(h);

    mpz\_clear(e);

    mpz\_clear(alpha);

    mpz\_clear(k);

    mpz\_clear(xC);

    mpz\_clear(yC);

    mpz\_clear(r);

    mpz\_clear(s);

    mpz\_clear(tmp);

    gmp\_randclear(state);

    return 0;

}

int GetUserKeys (mpz\_t xQ, mpz\_t yQ, int iteration, char\* login)

{

    FILE \*keys;

    char buffer[256];

    if ((keys = fopen("/usr/local/etc/sds/public\_accounts", "rb")) == NULL)

    {

        printf("Error reading accounts info\n");

        return -1;

    }

    if (feof(keys) == 0)

    {

        for (int i = 0; i<iteration; i++)

        {

            fgets(buffer, 256, keys);

            fgets(buffer, 256, keys);

            fgets(buffer, 256, keys);

        }

        if (feof(keys) != 0)

        {

            printf("End of file reached, no matches\n");

            return -1;

        }

        //узнать логин

        fgets(buffer, 256, keys);

    }

    else

    {

        printf("End of file reached, no matches\n");

        return -1;

    }

    for (int i=0; i<256; i++)

    {

        if (buffer[i] == 0x0A)

            buffer[i]=0;

        login[i]=buffer[i];

    }

    mpz\_inp\_str (xQ, keys, 16);

    mpz\_inp\_str (yQ, keys, 16);

    gmp\_printf("xQ = %Zx\n"

            "yQ = %Zx\n", xQ, yQ);

    fclose(keys);

    if (mpz\_cmp\_si(xQ, 0) > 0 && mpz\_cmp\_si(yQ, 0) > 0)

        return 0;

    else

    {

        printf("Public accounts file is corrupted\n");

        return -1;

    }

}

int CheckDS(mpz\_t p, mpz\_t a, mpz\_t b, mpz\_t m, mpz\_t q, mpz\_t xP, mpz\_t yP, mpz\_t xQ, mpz\_t yQ, FILE \*file)

{

    char result = 0;

    int keysfstat = 0;

    int it = 0;

    char login[256];

    mpz\_t e, alpha, nu, s, r, z1, z2, xC, yC, xC1, yC1,xC2, yC2, R;

    unsigned char \*h;

        printf("%x ", h[i]);

    printf("\n");

    //получить альфа, число, двоичным представлением которого является h

    mpz\_import(alpha, 32, 1, 1, 1, 0, h);

    gmp\_printf("alpha = %Zx\n", alpha);

    GetRSFromFile(file, r, s);

    gmp\_printf("r = %Zx\n", r);

    gmp\_printf("s = %Zx\n", s);

    //получить e

    mpz\_mod(e, alpha, q);

    if (mpz\_cmp\_si(e, 0) == 0)

        mpz\_set\_si(e, 1);

    gmp\_printf("e = %Zx\n", e);

    //получить ню

    Revers(q, e, nu);

    gmp\_printf("nu = %Zx\n", nu);

    //z1=s\*nu (mod q)

    mpz\_mul(z1, s, nu);

    mpz\_mod(z1, z1, q);

    //z2=-r\*nu (mod q)

    mpz\_mul(z2, r, nu);

    mpz\_neg(z2, z2);

    mpz\_mod(z2, z2, q);

    gmp\_printf("z1 = %Zx\n"

            "z2 = %Zx\n", z1, z2);

    PointMul(p, a, xP, yP, z1, xC1, yC1);

    while (result == 0)

    {

        keysfstat = GetUserKeys(xQ, yQ, it, login);

        {

            mpz\_clear(xC2);

            mpz\_clear(yC2);

            mpz\_clear(R);

            return -1;

        }

        it++;

        //C=z1\*P+z2\*Q

        PointMul(p, a, xQ, yQ, z2, xC2, yC2);

        PointSum(p, a, xC2, yC2, xC1, yC1, xC, yC);

        gmp\_printf("xC = %Zx\n"

                "yC = %Zx\n", xC, yC);

        mpz\_mod(R, xC, q);

        if (mpz\_cmp(R, r) == 0)

        {

            printf("Owner of this document is found!\n");

            printf("Owner is %s\n", login);

            result = 1;

        }

        else

            printf("Next try\n");

    }

    mpz\_clear(e);

    mpz\_clear(alpha);

    mpz\_clear(nu);

    mpz\_clear(r);

    mpz\_clear(z1);

    mpz\_clear(z2);

    mpz\_clear(xC);

    mpz\_clear(xC1);

    mpz\_clear(yC1);

    mpz\_clear(xC2);

    mpz\_clear(yC2);

    mpz\_clear(R);

    free(h);

    return 0;

}

int Verifier(const char \*file)

{

    int result = 0;

    mpz\_t p, a, b, m, q, xP, yP, xQ, yQ;

    mpz\_init(p);

    mpz\_init(a);

    mpz\_init(b);

    mpz\_init(m);

    mpz\_init(q);

    mpz\_init(xP);

    mpz\_init(yP);

    mpz\_init(xQ);

    mpz\_init(yQ);

    //статус открытого файла

    char parfstatus = 0;

    //получение параметров из файла

    parfstatus += GetParams("/usr/local/etc/sds/ds\_params", p, a, b, m, q, xP, yP);

    if (parfstatus == 0)

    {

        FILE \*target;

        //открытие целевого файла

        if ((target = fopen(file, "r")) == NULL)

        {

            printf("Error reading target file");

            Clear\_GMP(p, a, b, m, q, xP, yP, xQ, yQ);

            return -1;

        }

        else

        {

            fseek(target, 0, SEEK\_END);

            long fsize = ftell(target);

            fseek(target, 0, SEEK\_SET);

            if (fsize == 0)

            {

                printf("Empty file\n");

                Clear\_GMP(p, a, b, m, q, xP, yP, xQ, yQ);

                return -1;

            }

            else

            {

                //генерация подписи, добавление к файлу

                result += CheckDS(p, a, b, m, q, xP, yP, xQ, yQ, target);

            }

        }

    }

    //если параметры не получены

    else

    {

        printf("Get the parameters first!\n");

        return -1;

    }

    //очистить все переменные, используемые gmp

    Clear\_GMP(p, a, b, m, q, xP, yP, xQ, yQ);

    return result;

}

int execve(const char \*filename, char \*const argv[], char \*const envp[])

{

    FILE \*fp;

    struct stat buff;

    printf("\n%s\n", filename);

    if ((fp=fopen(filename, "rb"))==NULL)

    {

        printf("Unable to open file\n");

    }

    else

    {

        stat(filename, &buff);

        struct  passwd \*pw = getpwuid(buff.st\_uid);

        printf("uid: %s\n", pw->pw\_name);

        if (pw->pw\_uid != 0)

        {

            int result = Verifier(filename);

            if (result == -1)

            {

                char answer;

                while (answer != 'Y' && answer != 'y' && answer != 'N' && answer != 'n')

                {

                    printf("Are you sure you want to launch this application? Y/n ");

                    scanf("%c", &answer);

                    if (answer == 'N' || answer == 'n')

                        return NULL;

                }

            }

        }

        fclose(fp);

    }

    return ((orig\_execve)dlsym(RTLD\_NEXT, "execve"))(filename, argv, envp);

}
