ВВЕДЕНИЕ

В  современном  мире  организация  и  планирование  мероприятий  стали

неотъемлемой  частью  как  личной,  так  и  профессиональной  жизни.  С

развитием  цифровых  технологий  возрастает  потребность  в  удобных,

доступных  и  эффективных  системах  для  координации  событий,  управления

участниками  и  оптимизации  рабочего  процесса.  Особенно  важную  роль

системы  организации  и  планирования  мероприятий  играют  в  бизнесе,

образовании,  культуре  и  общественной  деятельности,  где  своевременное

информирование  и  четкая  организация  являются  залогом  успешного

проведения мероприятий.

Применение  клиент-серверных  приложений  для  планирования  и

организации мероприятий в бизнесе предоставляет ряд важных преимуществ.

Прежде  всего,

такие  приложения  позволяют  централизовать  всю

информацию  о  мероприятиях,  обеспечивая  быстрый  и  удобный  доступ  к

данным  для  всех  участников  процесса.  Автоматизация  задач,  таких  как

регистрация  участников,  бронирование  ресурсов,  рассылка  уведомлений  и

контроль за выполнением задач, значительно упрощает работу организаторов

и

снижает

вероятность

ошибок.  Клиент-серверные

приложения

способствуют  ускорению  коммуникации  внутри  команды  и  с  участниками

мероприятий,  обеспечивая  оперативное  распространение  информации  и

своевременное реагирование на изменения. Благодаря поддержке различных

форматов, офлайн, онлайн и гибридных мероприятий, компании могут гибко

адаптироваться  к

современным  условиям.  Системы  предоставляют

инструменты

аналитики  и

отчетности,

что  позволяет

оценивать

эффективность  мероприятий  и  выявлять  направления  для  их  улучшения.  В

результате применения таких решений компании экономят время и ресурсы,

повышают качество организации мероприятий и обеспечивают безопасность

обработки персональных данных участников.

Актуальность  данной  работы  обусловлена  тем,  что  требования  к

системам  планирования  и организации  мероприятий  растут.  В  современных

1.2.  Обзор существующих решений: преимущества и недостатки

Современные  клиент-серверные  приложения  для  планирования  и

организации  мероприятий  предлагают  широкий  спектр  функций,  таких  как

управление  регистрацией,

автоматизация  процессов,  интеграция

с

платежными  системами  и  аналитика.  Однако  они  также  имеют  свои

ограничения,  такие  как  высокая  стоимость,  сложности  в  кастомизации  и

зависимость  от  облачных  технологий.  Я  проанализировал  некоторые

существующие  клиент-серверные  приложения  в  области  планирования  и

организации  мероприятий  и  выявил  их  основные  преимущества  и

недостатки.

Cvent  —  это  облачная  платформа  для  управления  мероприятиями,

которая  помогает  компаниям  организовывать,  планировать  и  анализировать

события.

Преимущества Cvent:

1.

Полный  набор  инструментов  —  Cvent  охватывает  весь  процесс

управления  мероприятием:  от  регистрации  и  продажи  билетов  до  анализа  и

отчетности.

2.  Масштабируемость  —  подходит  для  мероприятий  любого

масштаба, от небольших встреч до крупных международных конференций.

3.

 Гибкость  в  настройках  —  позволяет  адаптировать  платформу

под специфические нужды каждого мероприятия.

4.

Интеграция  с  другими  системами  —  широкие  возможности

1.3.  Архитектура приложений для планирования мероприятий

Архитектура  клиент-серверных  приложений  представляет  собой

модель  организации  программной  системы,  при  которой  функции

распределяются  между  двумя  основными  компонентами  клиентом  и

сервером.  Эта  архитектура  основывается  на  сетевом  взаимодействии,  при

котором клиент отправляет запросы на выполнение определённых операций,

а сервер обрабатывает эти запросы, выполняет необходимую бизнес-логику и

возвращает результат клиенту.

Клиентская  часть  приложения  представляет  собой  компонент,

отвечающий  за  взаимодействие  с  пользователем.  Она  обеспечивает

отображение интерфейса, отправку запросов к серверу и обработку ответов.

В зависимости от типа платформы клиент может быть реализован в виде веб-

приложения, мобильного приложения или десктопной программы.

Основными  задачами  клиентской  части  являются  формирование  и

отображение  пользовательского  интерфейса  (например,  календаря  или

списка  мероприятий),  предоставление  функциональности  для  создания,

редактирования  и  просмотра  событий,  а  также  организация  сетевого

взаимодействия  с  серверной  частью  приложения.  Помимо  технической

составляющей,  важной  задачей  клиента  является  обеспечение  удобства  и

интуитивной  понятности  интерфейса,  что  напрямую  влияет  на  качество

пользовательского опыта.

Примеры технологий: React, Flutter, Android (Kotlin), iOS (Swift).

Серверная  часть  представляет  собой  центральный  компонент

программной системы, отвечающий за выполнение бизнес-логики, обработку

данных  и  обеспечение  взаимодействия  с  хранилищами  информации  и

внешними сервисами. Основной задачей сервера является приём и обработка

14

запросов,  поступающих  от  клиентских  приложений,  с  последующим

выполнением  необходимой  бизнес-логики,  включая  проверку  прав  доступа,

применение правил обработки событий и управление данными.

Сервер  осуществляет  постоянное  взаимодействие  с  базой  данных,  где

хранятся  все  ключевые  элементы  системы,  такие  как  сведения  о

пользователях,  мероприятиях,  приглашениях  и  напоминаниях.  Кроме  того,

сервер  может  выполнять  интеграцию  с  внешними  сервисами:  системами

отправки  уведомлений,  например,  по  электронной  почте  или  через  push-

уведомления,  очередями  задач  для  реализации  отложенной  обработки,  а

также сторонними API, такими как календарные сервисы, например, Яндекс

Календарь,  что  расширяет  функциональные  возможности  приложения  и

обеспечивает его интеграцию в более широкий цифровой контекст.

Примеры технологий: Node.js, Django, Spring Boot, FastAPI.

База  данных  представляет  собой  ключевой  компонент  программной

системы,  предназначенный  для  надёжного  хранения,  структурирования  и

управления

всей

информацией,

необходимой

для

полноценного

функционирования  приложения.  В  рамках  клиент-серверной  архитектуры

база  данных  выполняет  роль  централизованного  хранилища,  с  которым

взаимодействует

серверная  часть,  осуществляя  чтение,

запись  и

модификацию данных в ответ на запросы от клиентской части.

Хранимая  информация  охватывает  широкий  спектр  сущностей,

включая  учетные

записи  пользователей,  сведения  о  мероприятиях,

приглашениях, участниках, а также напоминания и уведомления. Эти данные

являются  основой  для  реализации  основной  функциональности  системы,

связанной  с  планированием  и  организацией  событий.  Таким  образом,  база

данных обеспечивает целостность и согласованность информации, а также её

доступность  для  всех  компонентов  приложения  в  процессе  обработки

запросов и выполнения бизнес-логики.

Примеры: PostgreSQL, MySQL, MongoDB.

15

Целостность  и  надёжность  функционирования

базы

данных

обеспечиваются  соблюдением  ACID-свойств,  которые  представляют  собой

совокупность четырёх ключевых принципов:  атомарности, согласованности,

изолированности и долговечности. Эти свойства являются основой надёжной

обработки транзакций в системах управления базами данных.

Атомарность

(Atomicity)

означает,

что

каждая

транзакция

выполняется как единое целое. Либо все её операции выполняются успешно,

либо, в случае ошибки, все изменения отменяются, и база данных остаётся в

исходном состоянии.

Согласованность  (Consistency)  гарантирует,  что  после  выполнения

транзакции база данных остаётся в корректном состоянии, соответствующем

всем установленным ограничениям, правилам и логике предметной области.

Каждая  транзакция  переводит  базу  из  одного  согласованного  состояния  в

другое.

Изолированность

(Isolation)

означает,

что

параллельно

выполняющиеся  транзакции  не  должны  влиять  друг  на  друга.  Изменения,

внесённые  одной  транзакцией,  становятся  видимыми  для  других  только

после  её  завершения,  что  позволяет  избежать  некорректного  чтения  или

обновления данных при одновременном доступе.

Долговечность (Durability) обеспечивает сохранность всех изменений,

внесённых  в  результате  успешно  завершённой  транзакции,  даже  в  случае

сбоев  оборудования  или  программного  обеспечения.  Это  достигается  путём

надёжной фиксации данных на постоянных носителях.

Соблюдение  ACID-свойств  критически  важно  для  обеспечения

корректной  и  предсказуемой  работы  базы  данных,  особенно  в  условиях

многопользовательского доступа и высокой нагрузки.

3.

взаимодействие  между  клиентом  и  сервером  через  использование

(GET,  POST,  PUT,  DELETE).  Этот

архитектурный  стиль  позволяет  упрощённо  организовать  обмен  данными

между  компонентами  приложения,  сохраняя  при  этом  гибкость  и

масштабируемость системы.

Серверная

часть

приложения

реализует  REST-интерфейс,

предоставляя  набор  HTTP-эндпоинтов,  через  которые  осуществляется

доступ  к  различным  данным  и  функционалу  системы.  Эти  эндпоинты

обрабатывают  запросы  от  клиента,  выполняют  необходимые  операции  и

возвращают ответы в формате JSON.

-

GET  —  используется  для  получения  данных  (например,  список

событий или детали пользователя).

-

POST  —  применяется  для  создания  новых  записей  (например,

создание нового события).

-

PUT  —  используется  для  обновления  существующих  данных

(например, изменение информации о событии).

-

DELETE  —  служит  для  удаления  данных  (например,  удаление

события).

Клиентская  часть,  реализованная  с  использованием  React,  направляет

HTTP-запросы к этим эндпоинтам, чтобы получать, изменять или удалять

информацию,  тем  самым  обеспечивая  взаимодействие  с  сервером  в  рамках

архитектуры REST.

Такой  подход  позволяет  изолировать  логику  обработки  данных  на

сервере,  а  на  клиенте  сосредоточиться  на  представлении  данных  и

взаимодействии с пользователем.

4.

Single Page Application (SPA).

Основной  целью  архитектуры  Single  Page  Application  (SPA)  является

улучшение пользовательского опыта через динамическую загрузку данных и

19

контента.  В  отличие  от  традиционных  приложений,  где  для  каждого

взаимодействия с сервером загружается новая страница, SPA загружает весь

интерфейс  только  один  раз,  а  последующие  взаимодействия  с  сервером

происходят  через  API.  Это  позволяет  избежать  полной  перезагрузки

страницы и делает приложение более быстрым и отзывчивым.

Клиентская часть реализовывается как Single Page Application (SPA) с

использованием  React.  Это  позволяет  динамически  обновлять  содержимое

интерфейса  без  необходимости  перезагрузки  страницы.  Все  данные,

необходимые  для  отображения,  запрашиваются  с  сервера  через  REST  API,

что  обеспечивает  плавную  работу  приложения,  без  задержек  при  переходах

между  различными  разделами,  быстрое  обновление  интерфейса  за  счёт

частичной загрузки данных, вместо полной перезагрузки, снижение нагрузки

на  сервер,  так  как  только  необходимые  данные  отправляются  и

обрабатываются без перезагрузки всей страницы.

В  результате,  пользователи  получают  более  интерактивное  и  быстрое

приложение, что способствует улучшению взаимодействия с системой.

5.

CORS и безопасность.

Основной  целью  настройки  CORS  (Cross-Origin  Resource  Sharing)

является  обеспечение  безопасности  взаимодействия  между  клиентом  и

сервером,  особенно  в  случаях,  когда  они  находятся  на  разных  доменах.

Например,  если  клиентская  часть  приложения  работает  на  localhost:3000,  а

серверная  —  на  localhost:8080,  необходимо  обеспечить  безопасный  обмен

данными  между  ними,  предотвратив  возможные  атаки  с  других,

несанкционированных источников.

CORS-фильтр  настраивается  на  серверной  стороне  для  контроля

доступа  при  выполнении  HTTP-запросов  с других доменов  или портов.  Это

позволяет:

-

ограничить  доступ  к  API-сервисам  только  для  разрешённых

источников (например, клиентская часть с localhost:3000);

20

-

исключить  возможность  выполнения  несанкционированных

запросов  с  чужих  доменов,  что  помогает  предотвратить  такие  атаки,  как

Cross-Site Request Forgery (CSRF) и Cross-Site Scripting (XSS);

-

повысить  уровень  безопасности  приложения,  гарантируя,  что

только клиент, который является доверенным источником, может обращаться

к серверным ресурсам.

Таким  образом,  правильная  настройка  CORS  играет  важную  роль  в

защите

данных

и

предотвращении

угроз,

связанных

с

несанкционированными запросами.

6.  Многослойная архитектура.

Многослойная  архитектура  предполагает  разделение  приложения  на

несколько  логических  слоёв,  каждый  из  которых  отвечает  за  свою

специфическую

задачу.

Это

разделение

помогает

улучшить

поддерживаемость,  расширяемость  и  тестируемость  системы,  а  также

способствует  упрощению  разработки  и  сопровождения.  В  типичной

многослойной  архитектуре  выделяют  три  основных  слоя:  представление

(UI), логика (сервис) и доступ к данным (DAO/repository).

Ключевые слои архитектуры и их функции:

1.

Контроллеры  (Controller)  —  обрабатывают  входящие  HTTP-

запросы  от  клиента,  выполняют  проверку  авторизации  и  прав  доступа,  а

также  передают  данные  в

соответствующий

слой  бизнес-логики.

Контроллеры  выступают  в  роли  посредников  между  клиентом  и

внутренними слоями приложения, управляя потоком данных.

2.

Сервисный слой (Service)  — инкапсулирует всю бизнес-логику

приложения.  Он  отвечает  за  выполнение  различных  операций,  таких  как

обработка  данных,  выполнение  вычислений,  вызовы  других  сервисов  или

интеграция  с  внешними  системами.  Сервисный  слой  обеспечивает

разделение  ответственности,  улучшая  читабельность  кода  и  упрощая  его

тестирование.

21

3.

Репозитории  (Repository)  —  осуществляют  взаимодействие  с

базой  данных,  предоставляя  абстракцию  для  работы  с  сущностями  через

JPA.  Репозитории  отвечают  за  выполнение  операций  CRUD  (создание,

чтение,  обновление,  удаление),  а  также  за  выполнение  сложных  запросов  к

базе  данных.  Этот  слой  позволяет  отделить  логику  взаимодействия  с

данными от бизнес-логики.

Таким  образом,  многослойная  архитектура  помогает  структурировать

приложение  таким  образом,  чтобы  каждая  его  часть  отвечала  за  чётко

определённые  функции.  Это  позволяет  эффективно  разрабатывать  и

поддерживать

приложение,

улучшать

масштабируемость

и

производительность, а также упрощать тестирование и сопровождение.

В  целом  использование  перечисленных  выше  подходов  обеспечивает

создание  стабильного,  структурированного  и  легко  масштабируемого  веб-

приложения,  в  котором  каждый  компонент  выполняет  чётко  определённую

функцию  и

взаимодействует

с  остальными  через  установленную

архитектурную модель.

Apache  Kafka  —  это  распределённая  платформа  для  публикации,

подписки,  хранения  и  обработки  потоков  данных.  Она  особенно  полезна  в

компонентами приложения.

Kafka  не  является  базой  данных  в  традиционном  смысле:  она  не

предназначена  для  долговременного  хранения  и  запроса  данных,  как

реляционные  или  NoSQL  СУБД.  Вместо  этого  она  часто  используется  в

качестве  промежуточного  слоя  между  источниками  данных  (например,

пользовательскими действиями или микросервисами) и системами хранения

(базами данных, аналитическими хранилищами и т. д.).

@Slf4j
@Service
@RequiredArgsConstructor
public class EventService {
    private final EventRepository eventRepository;
    private final EventMapper eventMapper;
    private final UserService userService;
    @Lazy
    private final ParticipantsService participantsService;

    // Получение мероприятия по ID
    public Event findById(Long id) {
        return eventRepository.findById(id);
    }

    // Создание нового мероприятия и назначение организатора
    public EventDto createEvent(Long userId, EventDto eventDto) {
        User user = userService.getUserById(userId);
        log.info("Creating new event by {}", user.getLogin());

        Event event = eventMapper.toEvent(eventDto);
        Event savedEvent = eventRepository.save(event);

        Participant participant = new Participant();
        participant.setUser(user);
        participant.setEvent(savedEvent);
        participant.setRole(UserEventRole.ORGANIZER);
        savedEvent.getParticipants().add(participant);

        log.info("Creating new event {}", savedEvent.getTitle());
        return eventMapper.toEventDto(eventRepository.save(savedEvent));
    }

    // Обновление существующего мероприятия
    @Transactional
    public EventDto updateEvent(EventDto eventDto, long eventId) {
        Event existingEvent = eventRepository.findById(eventId);
        log.info("Updating event: {} (id={})", existingEvent.getTitle(),

eventId);

        eventMapper.updateEventFromDto(eventDto, existingEvent);
        Event updatedEvent = eventRepository.update(existingEvent);

        log.info("Event {} updated successfully", updatedEvent.getId());
        return eventMapper.toEventDto(updatedEvent);
    }

    // Удаление мероприятия по ID
    @Transactional

    public void deleteEvent(Long eventId) {
        Event event = eventRepository.findById(eventId);
        log.info("Delete event: {} (id={})", event.getTitle(), eventId);
        eventRepository.delete(event);
    }

    // Получение мероприятия по ID пользователя
    public EventDto getEventById(Long userId) {
        Event event = eventRepository.findById(userId);
        return eventMapper.toEventDto(event);
    }

    // Получение мероприятия с участником по ID пользователя и

мероприятия

    public EventDto getEventByIdAndUserId(Long userId, Long eventId) {
        Event event = eventRepository.findById(userId);
        Participant participant =

participantsService.findParticipantByUserIdAndEventId(eventId, userId);

        event.setParticipants(List.of(participant));
        return eventMapper.toEventDto(event);
    }

    // Получение всех мероприятий пользователя с его ролями
    public List<EventDto> getEventsByUserIdWithRoles(Long userId) {
        List<Object[]> results =

eventRepository.findEventsWithRolesByUserId(userId);

        return results.stream()
                .map(result -> {
                    Event event = (Event) result[0];
                    UserEventRole role = (UserEventRole) result[1];

                    EventDto dto = eventMapper.toEventDto(event);
                    dto.setUserEventRole(role);
                    return dto;
                })
                .collect(Collectors.toList());
    }

    // Получение всех мероприятий пользователя
    public List<EventDto> getEventsByUserId(Long userId) {
        List<Event> events = eventRepository.findByUserId(userId);
        return eventMapper.toEventDtos(events);
    }
}
